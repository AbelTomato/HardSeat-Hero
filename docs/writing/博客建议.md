# 一、项目最初的问题定义

HardSeat Hero 的目标非常明确：

> 根据出发地、目的地和出行日期，查询低价火车直达与中转方案。

它不是抢票工具，也不是完整 12306 替代品。早期需求分析里刻意收敛了范围：

## 1. 必做范围

- 站点输入
- 日期输入
- 0 次或 1 次中转方案搜索
- 按最低总价排序
- 最短换乘时间过滤
- Mock 数据源
- FastAPI 查询接口
- React 前端展示
- 后端单元测试覆盖核心算法

## 2. 暂缓范围

- 用户登录
- 收藏方案
- 支付或购票跳转
- 自动抢票
- 多日价格趋势
- 2 次及以上中转
- 全国全量车站图谱搜索

这个收敛很关键。因为铁路路线搜索看起来像一个简单查询问题，实际上很快会遇到几个工程陷阱：

1. 官方票务数据不是稳定公开 API。
2. 真实票价和余票变化很快。
3. 一次中转会把一次查询放大成多组 OD 查询。
4. 全国站点全量枚举是不可接受的。
5. 如果目标是“最低价”，就不能随便用耗时、距离、历史价格做硬剪枝，否则可能漏掉真正便宜的路线。

所以项目一开始就定下了一个边界：

> 第一阶段不以官方实时票务数据为强依赖，先建立可插拔数据源、查询算法、后端 API 和前端展示闭环。

这个决策决定了整个项目后续架构。

# 四、第三阶段：真实票价数据源调研

对应开发日志：

```text
2026-06-25 票价获取方案调研
```

这个阶段的关键不是写代码，而是纠正数据源认知。

## 1. 被排除的方向

调研发现：

- `Parse12306` 可以提供车站、车次、经停时刻表思路，但不直接提供票价。
- `China-rail-way-stations-data` 主要是车站和车次始发终到数据，而且 README 标明数据过期，不提供票价。
- 用户提供的旧 `leftTicket/queryZ + queryTicketPrice` 链路存在问题：
  - 部分接口当前返回 HTML，不是 JSON。
  - 示例代码有席别分支 bug。
  - 余票字段不等于价格。
  - 交互式输入、全局状态、本地 CSV 方式不适合服务端。

## 2. 最终选择的真实票价接口

项目确认可行路径是 12306 官方公布票价页面使用的接口：

```text
GET https://kyfw.12306.cn/otn/leftTicketPrice/queryAllPublicPrice
```

链路：

```text
station_name.js
  ↓
站名 -> 电报码
  ↓
queryPublicIndex.html 建立上下文
  ↓
leftTicketPrice/queryAllPublicPrice
  ↓
queryLeftNewDTO
  ↓
TrainSegment / SeatPrice
```

已验证：

- 北京南到上海虹桥可返回 G/D 高铁动车票价。
- 北京到广州、上海到成都等 OD 可返回 K/Z 普速车硬座、硬卧、软卧等字段。

## 3. 票价字段解析

典型字段：

| 字段        | 席别     |
| ----------- | -------- |
| `swz_price` | 商务座   |
| `zy_price`  | 一等座   |
| `ze_price`  | 二等座   |
| `yz_price`  | 硬座     |
| `yw_price`  | 硬卧     |
| `rw_price`  | 软卧     |
| `gr_price`  | 高级软卧 |

价格编码以角为单位：

```text
07950 -> 795.0
02510 -> 251.0
```

后端用 `Decimal` 解析。

## 4. 这个阶段的判断

真实数据源不能直接侵入搜索服务。正确做法是：

```text
RouteSearchService
  -> TrainDataProvider
       -> Railway12306PublicPriceProvider
            -> StationCodeRepository
            -> PublicPriceClient
```

也就是说，12306 的 URL、header、字段名、异常类型，都不应该泄露到搜索算法。

# 六、第五阶段：真实数据源中转搜索的瓶颈暴露

接入真实 OD 查询后，项目遇到核心矛盾：

> 一次用户查询会被中转搜索放大成很多 OD 查询。

例如：

```text
候选站数 = 50
每个候选站最多查：
  A -> T
  T -> B

```

如果这些请求都打到 12306，问题很明显：

- 慢
- 容易失败
- 容易被限流
- 用户体验不可控
- 中转候选越多，远程 I/O 越不可控

这推动了后续几个方向：

1. 候选站剪枝
2. 请求预算
3. 并发控制
4. 缓存
5. 失败容错
6. 遥测
7. 静态票价库

# 七、第六阶段：中转搜索算法优化

对应提交：

```text
7a475d8  2026-06-26  feat(transfer): optimize route search candidates
902625e  2026-06-26  feat(search): add diagnostics and telemetry ranking
```

以及文档：

```text
docs/中转搜索算法优化路线.md
```

## 1. 候选中转站生成

项目没有选择“全国车站全量枚举”，而是引入 `CandidateTransferStationGenerator`。

候选站生成依据包括：

- 地理走廊过滤
- 投影方向过滤
- 铁路中心性
- 换乘枢纽和铁路通道先验
- 站名稳定排序

设计原则：

```text
先减少明显不合理的候选站，再把有限远程请求用在更可能便宜的候选上。
```

文档里明确指出：

> 地理距离不是铁路换乘成功率的核心因素。真正重要的是铁路拓扑、铁路局边界、停靠密度、历史命中率、车型覆盖和通道节点。

所以候选站排序从“纯几何距离”逐渐转向：

```text
换乘枢纽和铁路通道先验
铁路中心性
走廊垂距
站名
```

## 2. 一换搜索优化

早期组合逻辑是双层枚举：

```python
for first in first_legs:
    for second in second_legs:
        ...
```

复杂度是：

```text
O(n * m)
```

优化后：

- 第一段为空时，不再查询第二段。
- 第二段按发车时间排序。
- 对每个第一段，用二分定位可换乘的第二段。
- 加入最大总耗时过滤。
- 保持最低价主排序。

这减少了无效查询和组合成本。

## 3. 请求预算

真实 provider 必须有请求预算，例如：

```text
max_remote_queries
```

预算耗尽后：

- 不继续扩大外部请求
- 可以使用缓存
- 可以跳过候选
- 仍尽量返回已有直达或已有中转结果

这是从“算法正确”走向“系统可运行”的关键。

## 4. 受控并发

串行查询太慢，但无限并发会压垮数据源。所以新增：

```text
max_concurrent_remote_queries
```

用 `asyncio.Semaphore` 控制并发，用 `asyncio.Lock` 保护 `remote_query_count`，避免并发下超预算。

## 5. 单个候选失败不应中断整次搜索

真实网络下，某个 OD 查询失败很常见。项目调整为：

> 一换候选站单个 OD 请求失败时跳过该候选，不中断整次搜索。

这提升了结果鲁棒性。

# 八、第七阶段：最低价目标下的剪枝陷阱

这个项目里最值得写进博客的技术点之一是：

> 历史最低价通常是上界，不是安全 lower bound，不能直接用于 A* 硬剪枝。

## 1. 为什么这是问题

如果目标是最低价搜索，很容易想：

```text
如果历史上 A -> B 最低是 300 元，那么当前也至少要 300 元吧？
```

这个推断不安全。

历史观测价格可能高于当前真实最低价。也就是说历史最低价可能是：

```text
upper bound
```

而 A* 硬剪枝需要的是不会高估的：

```text
admissible lower bound
```

如果把历史价格当 lower bound，一旦它高估，就会错误剪掉未来可能更便宜的路线。

## 2. 项目的处理

项目引入了 `BestPriceTracker`，但只做安全剪枝：

- 直达方案先更新当前已知最低价。
- 如果第一段价格已经大于当前最佳价格，那么该中转不可能更低，可以跳过。
- 后来边界从 `>= best_price` 调整为 `> best_price`，以保留同价方案。

这体现了最低价搜索的一个核心原则：

> 可以用已知完整方案作为上界减少明显无效分支，但不能用不可靠历史价格作为硬 lower bound。

# 九、第八阶段：A* / Uniform Cost Search 与 Pareto 剪枝

对应开发日志：

```text
2026-06-26 中转搜索算法与真实数据源稳定性优化
```

项目开始支持 `max_transfers >= 2` 的路线搜索。

## 1. CheapestPathSearcher

新增：

- `PathSearchState`
- `CheapestPathSearcher`

状态包含：

- 当前站
- 已选车次
- 累计票价
- 到达时间
- 已访问站点

使用优先队列按：

```text
g + h
```

扩展路径。

但当前 `_heuristic_lower_bound()` 返回 `0`，所以算法实际上是：

```text
Uniform Cost Search
```

这又是一个保守设计：

> 先保证最低价正确性，再考虑更激进的启发函数。

## 2. Pareto frontier

新增 `ParetoFrontier`，用于状态支配剪枝。

状态只有在已有状态满足：

```text
累计价格不高
且
到达时间不晚
```

时才会被剪枝。

这避免了一个常见错误：

> 更贵但更早到达的状态，可能接上更便宜的后续车次，不能简单删除。

这部分适合博客中单独写成“最低价路径搜索不能只看价格”的小节。

# 十二、第十一阶段：结果体验排序与同价方案处理

对应开发日志：

```text
2026-06-26 最低价结果体验排序与后续瓶颈判断
```

项目在这一步意识到：

> 最低价是主目标，但同价方案之间仍需要更符合用户直觉的排序。

## 1. 同价排序

排序策略调整为：

```text

2. 中途等待时间舒适性惩罚
3. 换乘次数
4. 总耗时
```

等待时间舒适性：

- `30~120` 分钟：舒适，惩罚为 0
- 小于 30 分钟：过短，风险高
- 超过 120 分钟：过长，惩罚增加

这不会改变最低价目标，只影响同价方案顺序。

## 2. 同价方案保留

剪枝边界从：

```text
>= best_price
```

```text
> best_price
```

## 3. 方案去重

```text

```

用于去重，避免同一车次组合因日期窗口查询重复进入结果。

## 4. 结果数量限制

新增测试确保最多返回 10 个方案。

这说明项目开始从“算法能跑”进入“结果可读、可解释、不过载”的阶段。

# 十三、第十二阶段：瓶颈判断转向静态票价库

对应开发日志：

```text
2026-06-26 静态票价数据库方案
```

项目做了一个重要判断：

> 短期不应继续主要微调本地搜索循环，而应优先建立历史/静态票价持久化数据库，用稳定价格数据减少真实 Provider 的远程 I/O。

原因很直接：

- 真实数据源瓶颈主要是远程 I/O。
- 公布票价相对稳定。
- 用户搜索通常集中在热门 OD。
- 中转候选也可以从历史命中中推导。
- 本地数据库查询比实时请求 12306 稳定得多。

## 1. 静态票价库的产品语义

它提供的是：

```text
公布票价 / 历史观测最低票价路径
```

不提供：

```text
实时余票保证
实时停运保证
动态折扣保证
```

这一区分必须写清楚。否则用户会误以为它是实时购票结果。

## 2. 为什么用 SQLite

第一阶段选择 SQLite：

- 项目当前是本地 MVP
- 零运维
- 读多写少
- 与已有缓存和遥测技术栈一致
- 便于本地演示和测试

后续如果多人共享、多进程写入、中心化服务，再迁移 PostgreSQL。

## 3. 不做全国 OD 暴力爬取

全国车站 OD 是 `N²` 规模。暴力枚举会导致：

- 请求量巨大
- 大量无效 OD
- 触发限流
- 爬取周期不可控
- 数据库膨胀但命中率未必高

所以采用分层采集：

1. 热门 OD
2. 用户查询 OD
3. 候选中转 OD
4. 线路/车次级准全量

# 十九、博客可以突出的几个技术判断

## 1. 先 Mock，再真实数据源

不要一开始就把系统绑死在 12306。真实数据源不稳定，Mock provider 是开发、测试和演示的安全基线。

## 2. Provider 抽象是核心资产

`RouteSearchService` 不知道数据来自哪里。它只处理统一的 `TrainSegment`。

这让项目能从：

```text
Mock
```

平滑演进到：

```text
12306 provider
```

再到：

```text
SQLite static provider
```

再到：

```text
static + remote fallback
```

## 3. 最低价搜索不能乱剪枝

历史最低价不是安全 lower bound。它可以用于：

- 排序
- 优先级
- 刷新队列
- 展示

但不能直接作为 A* 硬剪枝依据。

这是项目里最有深度的算法经验之一。

## 4. 远程 I/O 比本地算法更早成为瓶颈

当真实 provider 接入后，最大问题不是 Python 循环慢，而是：

```text
中转搜索放大远程 OD 请求
```

所以正确优化方向是：

- 缓存
- 请求预算
- 候选站剪枝
- 静态票价库
- 遥测反哺刷新

而不是无限微调内存里的组合循环。

## 5. 静态票价库改变系统形态

项目从“实时请求器”变成了：

```text
本地静态数据优先 + 远程补洞 + 遥测驱动刷新
```

这是从原型向可用系统演进的关键一步。

## 6. 前端诊断不是花哨功能

当数据源不稳定时，用户只看到“无结果”是不够的。诊断面板能解释：

- 当前 provider
- 请求预算
- 远程请求数
- 缓存命中
- 失败原因
- 候选站扩展情况

这对开发调试和用户信任都重要。

# 二十三、写作角度建议

建议不要写成流水账“我做了哪些功能”。更适合写成“问题驱动的演进”：

```text
问题 1：没有真实数据怎么办？
解决：Mock + Provider 抽象。

问题 2：真实接口不稳定怎么办？
解决：错误包装、默认 Mock、受控失败。

问题 3：中转搜索请求太多怎么办？
解决：候选站剪枝、请求预算、并发上限、缓存。

问题 4：最低价搜索怎么保证不误剪？
```
