# 中转搜索算法优化路线

## 目标定位

当前项目不应只定位为“12306 中转查询”。更准确的产品目标是：

> 全国铁路最低票价路径搜索器（Cheapest Railway Route Planner）

核心目标函数只有一个：**总票价最低**。

总耗时、换乘等待时间、换乘次数、车站舒适度等指标仍然保留，但只作为同价方案的稳定排序条件或展示信息，不应干扰最低票价搜索。

## 当前实现位置

| 模块         | 文件                                                                              | 当前职责                                                    |
| ------------ | --------------------------------------------------------------------------------- | ----------------------------------------------------------- |
| 路线搜索     | `backend/app/services/route_search.py`                                            | 查询直达、生成一换方案、组合车次、排序返回结果              |
| 候选中转站   | `backend/app/services/transfer_candidates.py`                                     | 基于地理走廊、低价 hub、中心性生成候选站                    |

| 12306 数据源 | `backend/app/adapters/railway_12306_public_price.py`                              | 调用公开票价接口，并接入候选站生成器                        |

| 现有测试     | `backend/tests/test_route_search.py`、`backend/tests/test_transfer_candidates.py` | 覆盖基础搜索、缓存、预算、候选站逻辑                        |

## 当前流程

```text
search(query)
    ├── 查询直达 A -> B
    └── _iter_one_transfer(query)
            ├── candidate_transfer_stations(query)
            ├── 查询第一段 A -> T
            ├── 查询第二段 T -> B
            └── first_legs × second_legs 两两组合
```

当前实现已经具备几个正确方向：

- 有候选站剪枝，避免全站图暴力枚举。
- 有 TTL 缓存，重复请求可以复用结果。
- 有流式返回接口，便于后续边搜索边产出。
- 候选站生成使用地理投影和走廊过滤，比纯距离排序更合理。
- 模块边界清晰，路线搜索和候选站生成可以独立演进。

## 当前主要瓶颈

### 1. 候选站串行查询

当前 `_iter_one_transfer()` 对候选站逐个处理：

```text
候选站数 × 2 次远程查询
```

如果候选站为 50 个，理论上需要最多 100 次远程查询。即使缓存可以降低重复查询成本，首次查询仍然昂贵。

当前查询是串行 IO：

```text
查广州 -> 等待 -> 查长沙 -> 等待 -> 查南昌 -> 等待
```

这会把网络延迟线性累加。

### 2. 第一段为空仍可能浪费策略空间

一换方案必须满足：

```text
A -> T 存在车次
T -> B 存在车次
```

如果第一段 `A -> T` 没有车，则不需要查询第二段 `T -> B`。当前结构虽然先写第一段再写第二段，但策略上没有显式把“第一段为空直接剪枝”作为优化原则，也没有围绕它设计测试。

### 3. 车次组合是 O(n*m)

当前组合逻辑是：

```python
for first in first_legs:
    for second in second_legs:
        ...
```

如果第一段 20 趟、第二段 20 趟，一个中转站就是 400 次组合判断。50 个候选站就是 20000 次组合判断。

车次天然有时间顺序，不需要暴力两两组合。可以将第二段按 `depart_at` 排序，然后对每个第一段用二分查找找到满足最小换乘时间的第一个第二段，复杂度从 `O(n*m)` 降到 `O(n log m)`；如果两边都按时间排序，也可以用双指针做到 `O(n+m)`。

### 4. `remote_query_interval_seconds` 不适合作为并发限流核心

当前 `_search_segments()` 内部使用：

```python
await asyncio.sleep(remote_query_interval_seconds)
```

这对串行请求有意义，但对并发请求不能提供可靠限流。改成并发后，应使用：

```python
asyncio.Semaphore(max_concurrent_remote_queries)
```

`max_remote_queries` 继续作为总预算，`Semaphore` 作为实时并发预算，sleep 只可作为兼容性降级选项。

### 5. 候选站评分仍偏几何

当前候选站评分主要是：

```text
是否低价 hub
垂直距离
中心性
站名
```

地理距离不是铁路换乘成功率的核心因素。真正重要的是铁路拓扑、铁路局边界、停靠密度、历史命中率和低价断点概率。

例如同一条走廊上，长沙通常比湘潭更有中转价值，因为长沙的车次覆盖和停靠密度更高。

### 6. 启发式只用于筛候选，没有用于控制搜索顺序

当前做法是：

```text
先生成 50 个候选站
再全部尝试
最后排序
```

这不是严格意义上的启发式搜索。真正有效的搜索应当是：

```text
优先尝试估计总价最低的候选
找到当前 best_price 后
剪掉所有理论上不可能更便宜的分支
```

这就是 Best First Search / A* 的方向。

## 优化原则

### 1. 目标函数单一化

搜索阶段只优化价格：

```python
plans.sort(key=lambda plan: (plan.total_price, plan.total_duration_minutes, len(plan.transfer_stations)))
```

可以保留总耗时和换乘次数作为同价兜底排序，但主目标必须是 `total_price`。

当前体验策略进一步明确为：

```text
总价最低优先
同价时按中途等待时间舒适性排序
最终最多保留 10 个方案
```

等待时间舒适性第一版只考虑每次换乘等待：

- `30~120` 分钟：舒适。
- 小于 `30` 分钟：过短，风险高。
- 大于 `120` 分钟：过长，体验差。

### 2. 先剪远程查询，再优化本地组合

远程查询比 Python 组合更贵。优化优先级：

1. 少查无效区间。
2. 控制并发提高 IO 利用率。
3. 用价格下界剪枝。
4. 再优化本地组合复杂度。

当前判断应进一步前移：优先建立可复用的历史最低价持久化层。对于真实数据源，减少远程 OD 请求次数通常比继续优化本地组合循环收益更高。

### 3. 保持可证明正确的价格剪枝

剪枝条件不能丢掉真实最低价。价格剪枝必须基于不会高估的 lower bound：

```text
已累计票价 + 剩余最低票价下界 >= 当前 best_price
```

如果 lower bound 可能高估，就可能误剪最优解，不能用于 A* 的 admissible heuristic。

## 分阶段演进路线

### 阶段 1：一换搜索的低风险优化

目标：在不改变外部接口的前提下，降低无效查询和组合成本。

改动文件：

- `backend/app/services/route_search.py`
- `backend/tests/test_route_search.py`

核心改动：

1. 在 `_iter_one_transfer()` 中显式执行：第一段为空则跳过第二段。
2. 第一段和第二段过滤掉 `lowest_price is None` 的车次。
3. 车次按时间排序。
4. 使用二分或双指针匹配满足 `min_transfer_minutes` 的第二段。
5. 保持 `max_total_duration_minutes` 过滤。
6. 返回排序保持价格优先，耗时和换乘次数只作为稳定排序条件。

验收标准：

- 第一段无车时不会查询第二段。
- 同价时输出顺序稳定。
- 原有缓存测试和预算测试不回退。
- 一换结果仍能覆盖现有 mock 场景。

### 阶段 2：受控并发查询

目标：提升候选站搜索速度，同时避免压垮数据源。

改动文件：

- `backend/app/services/route_search.py`
- `backend/app/api/routes.py`
- `backend/tests/test_route_search.py`

核心设计：

```python
RouteSearchService(
    provider,
    max_remote_queries=120,
    max_concurrent_remote_queries=5,
)
```

并发策略：

```text
候选站任务并发调度
每个真实远程查询必须获取 Semaphore
max_remote_queries 控制总次数
segment_cache 命中不占用远程预算
```

注意点：

- `remote_query_count` 在并发环境下需要避免竞态；可用 `asyncio.Lock` 保护预算检查和递增。
- 缓存读写也要避免重复击穿；短期可以接受，长期应引入 singleflight。
- `remote_query_interval_seconds` 不再作为主限流手段。

验收标准：

- 并发数不超过配置值。
- 总远程查询数不超过 `max_remote_queries`。
- 缓存命中不增加 `remote_query_count`。
- mock 测试可稳定复现。

### 阶段 3：价格优先剪枝

目标：让已知低价方案反向减少后续搜索。

核心状态：

```text
best_price: Decimal | None
```

一换场景剪枝：

```text
如果 first.lowest_price >= best_price
则该 first 不可能产生更低总价
```

进一步引入保守下界：

```text
first.lowest_price + lower_bound(T -> B) >= best_price
则跳过第二段或跳过组合
```

初始 lower bound 可以使用 `0`，这不会误剪，但剪枝能力弱。后续可来自历史最低价表。

验收标准：

- 不丢失最低价方案。
- 当已有低价直达时，明显减少无效中转组合。
- 价格相同的方案仍可保留或按策略截断。

### 阶段 4：候选站评分升级

目标：从“地理走廊候选”升级为“低价断点候选”。

改动文件：

- `backend/app/services/transfer_candidates.py`
- `backend/tests/test_transfer_candidates.py`

建议评分：

```text
score = (
    -cheap_transfer_score,
    -railway_centrality_score,
    perpendicular_distance,
    projection_penalty,
    station_name,
)
```

候选站特征：

| 特征                   | 含义                             |
| ---------------------- | -------------------------------- |
| `cheap_transfer_score` | 低价断点概率，优先级高于距离     |
| `centrality_score`     | 停靠密度和铁路枢纽程度           |
| `perpendicular`        | 偏离主走廊距离                   |
| `projection`           | 是否位于起终点之间或合理延长线上 |
| `historical_hit_rate`  | 历史搜索中成为最低价路径的比例   |

第一版不需要机器学习。可以先维护静态表：

```text
赣州、龙岩、长沙、南昌、郑州、徐州、广州、衡阳、株洲、南京、济南、天津
```

验收标准：

- 典型长距离查询优先返回铁路枢纽和低价断点。
- 地理距离只作为过滤条件，不作为唯一排序核心。
- 未知站点仍有 fallback 结果。

### 阶段 5：多级缓存

目标：减少重复解析和重复远程查询。

当前缓存 key：

```text
provider_name:date:from_station:to_station
```

建议拆分：

| 缓存层           | TTL         | 内容                               |
| ---------------- | ----------- | ---------------------------------- |
| 车次结构缓存     | 24h 或更长  | 车次号、起终站、经停车站、时间结构 |
| 票价缓存         | 30min       | 席别价格                           |
| 余票缓存         | 1min 或实时 | 余票状态                           |
| 候选站缓存       | 24h         | 查询区间对应候选站列表             |
| 历史 lower bound | 长期        | `from -> to` 历史最低票价          |

短期可以继续使用内存 TTL。长期应迁移到 Redis，支持跨进程复用。

验收标准：

- 同一区间重复查询不重复解析稳定车次结构。
- 票价和余票可以独立过期。
- 不因缓存过期策略导致票价明显陈旧。

### 阶段 5A：历史最低价持久化数据库

目标：用稳定的历史最低价降低真实数据源 I/O 成本，并为候选排序提供依据。

第一版表结构建议：

```sql
CREATE TABLE IF NOT EXISTS od_min_prices(
    provider TEXT NOT NULL,
    origin TEXT NOT NULL,
    destination TEXT NOT NULL,
    min_price TEXT NOT NULL,
    observed_at TEXT NOT NULL,
    updated_at TEXT NOT NULL,
    PRIMARY KEY(provider, origin, destination)
);
```

后续可扩展字段：

```text
train_type
seat_type
travel_date_bucket
source_query_date
sample_count
```

写入时机：

1. 每次 `_search_segments()` 从 Provider 或 OD 缓存拿到有价车次后，提取该 OD 的最低 `lowest_price`。
2. 如果新价格低于已有 `min_price`，更新最低价。
3. 如果价格未降低，只更新 `observed_at` 或样本统计，避免频繁覆盖稳定最低价。

读取用途：

| 用途                         | 是否允许 | 说明                                                    |
| ---------------------------- | -------- | ------------------------------------------------------- |
| 候选 OD 排序                 | 允许     | 低历史最低价优先查询。                                  |
| 候选中转站排序               | 允许     | `A -> T` 与 `T -> B` 历史价更低的中转站优先。           |
| 展示历史最低价更新时间       | 允许     | 向用户说明价格新鲜度。                                  |
| 直接跳过远程查询并返回历史价 | 谨慎     | 只有用户接受非实时价格时才可做。                        |
| A* 硬剪枝 lower bound        | 默认禁止 | 历史最低价可能高于当前真实最低价，不能保证 admissible。 |

服务边界建议：

```text
RouteSearchService
    ├── SegmentQueryService
    │       ├── 内存 TTL 缓存
    │       ├── SQLite OD 车次缓存
    │       ├── OD 历史最低价库
    │       └── Provider 远程查询
```

验收标准：

- 查询到有价 OD 后会写入或更新历史最低价。
- 重复搜索时可读取历史最低价用于排序或诊断。
- 历史最低价记录包含更新时间。
- 未启用数据库时搜索行为不变。
- 不因历史最低价误剪严格最低价方案。

### 阶段 6：历史命中率与价格下界

目标：让系统越来越偏向低价成功路径。

需要记录：

```text
query_origin
query_destination
transfer_station
found_plan_price
is_best_plan
search_timestamp
```

可派生指标：

| 指标                                   | 用途             |
| -------------------------------------- | ---------------- |
| `historical_hit_rate`                  | 候选站排序       |
| `historical_min_price(from, to)`       | A* lower bound   |
| `station_min_departure_price(station)` | 粗粒度价格剪枝   |
| `transfer_success_rate`                | 减少低成功率候选 |

重要约束：

- 用于 A* 剪枝的 lower bound 必须不能高估。
- 如果历史最低价不是严格下界，只能用于排序，不能用于安全剪枝。

验收标准：

- 历史数据缺失时行为退化为静态启发式。
- 历史数据存在时，候选站顺序可被稳定影响。
- 不因历史数据误剪真实最低价。

### 阶段 7：Best First Search

目标：把搜索从“遍历候选站”改为“按估计总价扩展最有希望的状态”。

一换版本可以先不支持任意 K 次中转，只替换搜索顺序。

状态：

```text
State(
    current_station,
    segments,
    accumulated_price,
    transfer_count,
    earliest_depart_or_arrive_time,
)
```

优先级：

```text
priority = accumulated_price + heuristic_lower_bound(current_station, destination)
```

搜索过程：

```text
初始化：A -> B 直达状态 / A -> T 候选扩展
弹出 priority 最低的状态
如果到达 B，更新 best_price
如果 priority >= best_price，停止或剪枝
否则扩展下一段
```

验收标准：

- 在一换场景下结果不差于旧实现。
- 搜索顺序由估计价格控制，而不是候选站列表顺序控制。
- 找到足够低的 `best_price` 后，可以提前停止后续高估价候选。

### 阶段 8：A* 最终形态

目标：统一支持 0 次、1 次、2 次乃至 K 次中转。

核心模型：

```text
图节点：车站
边：某日期窗口内 from -> to 的可乘车次
边权：最低票价
约束：发到时间、最小换乘时间、最大总时长、最大中转次数
目标：从起点到终点的最低总票价路径
```

A* 状态：

```text
station: 当前站
segments: 已选车次列表
g: 已累计票价
h: 到终点的最低票价下界
f: g + h
arrive_at: 当前路径最后到达时间
transfer_count: 已用中转次数
```

启发函数：

```text
h(station, destination) = lower_bound_price(station, destination)
```

可选 lower bound 来源：

1. `0`：绝对安全，但剪枝弱。
2. 历史最低票价：如果保证不会高估，可以安全剪枝。
3. 距离 × 理论最低每公里票价：通常可能高估或低估，需要谨慎，只建议排序，不建议硬剪。
4. 站点最低出发价：可用于粗排，不一定可用于严格剪枝。

终止条件：

```text
如果 priority_queue 最小 f >= best_price
则不存在更低价路径，可以停止
```

剪枝条件：

```text
transfer_count > max_transfers
total_duration > max_total_duration_minutes
next_depart_at < arrive_at + min_transfer_minutes
g + h >= best_price
同一 station、transfer_count、时间桶 下已有更低 g
```

验收标准：

- 支持 `max_transfers = 0/1/2`，且行为可解释。
- A* 在 `h=0` 时退化为 Uniform Cost Search，仍保证最低价。
- 使用 admissible heuristic 时保证不丢最优解。
- 可通过 mock 图构造证明：当存在更便宜两换方案时，能优先找到并返回。

### 任务 2：加入价格剪枝

- 在搜索过程中维护 `best_price`。
- 对明显不可能优于当前最低价的第一段进行剪枝。
- 保证同价方案处理策略明确。

### 任务 3：加入受控并发

- 增加 `max_concurrent_remote_queries` 配置。
- 用 `asyncio.Semaphore` 包住真实远程查询。
- 用 `asyncio.Lock` 保护 `remote_query_count`。

### 任务 4：候选站评分重构

- 增加静态低价断点权重。
- 让 `cheap_transfer_score` 优先于几何距离。
- 更新候选站测试。

### 任务 5：设计 A* 搜索器

- 新增独立搜索器，不直接把 `_iter_one_transfer()` 改成复杂状态机。
- 先实现 `h=0` 的 Uniform Cost Search。
- 再引入安全 lower bound。
- 最后支持 `max_transfers = 2`。

### 任务 6：实现 OD 历史最低价数据库

- 新增历史最低价存储模块，例如 `backend/app/services/price_history.py`。
- 使用 SQLite 保存 `provider/origin/destination/min_price/updated_at`。
- 在 OD 查询得到有价车次后更新历史最低价。
- 在候选排序或诊断中读取历史最低价。
- 补充单元测试：新增、更新、不覆盖更低价格、更新时间、未启用时行为不变。

## 最终架构建议

```text
RouteSearchService
    ├── DirectRouteSearcher
    ├── TransferCandidateProvider
    ├── SegmentQueryService
    │       ├── Semaphore 限流
    │       ├── 多级缓存
    │       └── Provider 适配
    ├── CheapestPathSearcher
    │       ├── Best First Search
    │       ├── A* heuristic
    │       └── Dominance pruning
    └── SearchTelemetryRecorder
            ├── 历史命中率
            ├── 历史最低价
            └── 候选站反馈
```

这样可以避免 `RouteSearchService` 继续膨胀，同时为后续 Redis、历史学习和多次中转留出边界。

## 结论

当前实现的方向是正确的，但它仍然是“候选站枚举 + 最后排序”的结构。真正的最低票价搜索应升级为“价格驱动的启发式路径搜索”。

短期应先完成一换搜索的确定性优化：少查、并发、低复杂度组合、价格剪枝。

长期应引入 Best First Search / A*：用优先队列控制搜索顺序，用 lower bound 安全剪枝，用历史数据优化候选和启发函数。最终目标不是复刻 12306 推荐，而是构建一个以最低票价为唯一主目标的铁路路径规划引擎。
