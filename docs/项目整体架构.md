# 项目整体架构

## 1. 项目定位

HardSeat Hero 是一个前后端分离的低价火车路线查询本地 MVP。用户输入出发站、到达站和出行日期后，系统返回直达或中转方案，并以最低总票价作为主排序目标。

项目已经从纯 Mock 原型演进为包含真实数据源适配、SQLite 静态票价库、OD 缓存、搜索遥测、刷新脚本和诊断脚本的本地系统。当前架构重点是：

1. 降低远程 12306 公布票价接口 I/O。
2. 通过静态票价库提升常见 OD 命中率。
3. 用搜索遥测反哺刷新 OD 队列。
4. 保持搜索正确性，不把历史最低价当作安全 lower bound。

明确不做：

- 用户登录、收藏、支付、购票、抢票。
- 全国 OD 暴力采集。
- 把静态公布票价等同于实时余票。
- 把长周期刷新任务放入 FastAPI 请求链路。
- 把历史最低价作为 A* 硬剪枝 lower bound。

## 5. 前端请求链路

### 5.1 站点自动补全

```text
用户输入站名
  ↓
App.tsx useEffect
  ↓
searchStations(query)
  ↓
GET /api/stations/search?q=...
  ↓
后端 provider.search_stations()，如果 provider 未实现则返回空列表
  ↓
StationSearchResponse
  ↓
datalist 展示候选站点
```

Mock provider 使用内置站点列表；12306 provider 通过站点 JS 解析站名到电报码映射；static-price provider 当前不承担完整站点库能力。

### 5.2 流式路线查询

前端主查询走流式接口：

```text
用户提交查询表单
  ↓
App.tsx handleSubmit()
  ↓
searchRoutesStream(payload, onPlan, signal)
  ↓
POST /api/routes/search/stream
  ↓
ReadableStream + TextDecoderStream
  ↓
逐行消费 NDJSON
  ↓
增量更新 result.plans
  ↓
刷新结果列表和诊断面板
```

流式返回的每一行是一个 `TransferPlan` JSON；如果后端捕获到 `Railway12306Error`，会返回带 `error` 字段的 JSON 行，前端转换为 `ApiError(data_source_unavailable)`。

注意：前端当前固定提交 `max_transfers=1`。后端模型支持 `0..2`，二换能力主要通过 API 或后续前端扩展使用。

### 5.3 数据源状态查询

```text
App.tsx 初始化或查询完成
  ↓
getProviderStatus()
  ↓
GET /api/providers/status
  ↓
显示 provider、请求预算、并发上限、缓存命中、剪枝次数、候选扩展情况
```

```

## 7. 领域模型

文件：`backend/app/domain/models.py`。

| 模型 | 职责 |
| --- | --- |
| `SeatPrice` | 席别、价格、余票状态占位 |
| `TrainSegment` | 单段车次，包含车次号、起终站、时刻、历时、价格、来源和更新时间 |
| `RouteQuery` | 查询条件：出发站、到达站、日期、最大中转次数、最短换乘时间、最大总耗时 |
| `TransferPlan` | 完整路线方案，包含总价、总耗时、换乘等待、换乘站和车次段 |
| `RouteSearchResponse` | 路线查询响应 |
| `StationSearchResponse` | 站点补全响应 |
| `StationMetadata` | 候选中转站元数据 |

关键约束：

- `TrainSegment.arrive_at` 必须晚于 `depart_at`。
- `RouteQuery.max_transfers` 限制为 `0..2`。
- `SeatPrice.price` 使用 `Decimal`，避免浮点误差。

文件：`backend/app/adapters/static_price_fallback_provider.py`。

  ├─ 命中：直接返回

          ↓
       远程返回车次
          ↓

          ↓

```

## 9. 搜索服务架构

核心文件：`backend/app/services/route_search.py`。

### 9.1 总体搜索链路

```text
RouteSearchService.search(query)
  ↓
_reset_search_state()
  ↓
_iter_plans(query)
  ├─ max_transfers < 2：直达 + 一换搜索
  └─ max_transfers >= 2：CheapestPathSearcher
  ↓
_deduplicate_plans()
  ↓
按 _plan_sort_key() 排序
  ↓
截断到 MAX_RETURNED_PLANS = 10
  ↓
_record_search()
  ↓
RouteSearchResponse
```

`stream(query)` 使用相同 `_iter_plans()`，逐个产出当前进入前 10 的 `TransferPlan`，结束时记录遥测。

### 9.2 一换搜索

一换搜索使用候选中转站，不枚举全国站点：

```text
provider.candidate_transfer_stations(query)
  ↓
查询 A -> T
  ↓
若第一段无结果，剪枝，不查 T -> B
  ↓
查询 T -> B
  ↓
按第二段出发时间排序
  ↓
二分定位满足最短换乘时间的第二段
  ↓
组合可行方案
  ↓
按 best_price、总耗时等条件剪枝
```

直达搜索会先建立已知最低价上界 `best_price`，用于后续中转剪枝。

### 9.3 二换搜索

当 `RouteQuery.max_transfers >= 2` 时启用 `CheapestPathSearcher`。当前实现是 A* 框架，但启发函数返回 `0`：

```python
def _heuristic_lower_bound(self, from_station: str, to_station: str) -> Decimal:
    return Decimal("0")
```

因此当前等价 Uniform Cost Search。历史最低票价只允许用于排序、刷新优先级和展示，不允许作为硬剪枝 lower bound。

### 9.4 Pareto frontier

`ParetoFrontier` 以 `(当前站, 中转次数)` 为 key。只有已有状态同时满足：

```text
已有累计价格 <= 当前累计价格
且已有到达时间 <= 当前到达时间
```

时才剪枝。这样保留“更贵但更早到达”的状态，避免误剪可能形成低价后续路径的方案。

### 9.5 排序规则

主排序目标始终是最低总价：

```text
1. total_price
2. 换乘等待舒适性惩罚
3. 换乘次数
4. 总耗时
```

换乘等待舒适区间为 `30~120` 分钟。等待过短或过长只影响同价方案排序，不改变最低价主目标。

### 9.6 请求预算、并发和缓存链路

`_search_segments()` 查询顺序：

```text
1. 内存 TTL 缓存
2. SQLite OD 持久化缓存（仅当 ROUTE_SEGMENT_CACHE_DB 启用）
3. provider
4. 写回内存缓存
5. 写回 SQLite OD 缓存（若启用）
```

并发控制：

- `max_remote_queries`：单次搜索最大 OD 请求数。
- `max_concurrent_remote_queries`：provider 查询并发上限。
- `remote_query_interval_seconds`：非 Mock provider 默认请求间隔。

诊断信息保存在 `SearchDiagnostics`：远程请求数、内存缓存命中、SQLite 缓存命中、已扩展候选、失败候选、best_price 剪枝和 Pareto 剪枝次数。

## 10. 候选中转站生成

文件：`backend/app/services/transfer_candidates.py`。

核心对象：`CandidateTransferStationGenerator`、`StationMetadataRepository`、`CandidateTransferConfig`。

逻辑：读取出发站和到达站元数据，计算 OD 球面距离，按 corridor 筛选走廊内站点，剔除过近端点站，按低价断点评分、中心性、走廊偏离等排序，返回 Top N 候选站。站点元数据缺失时 fallback 到低价枢纽列表。

## 11. 缓存与持久化

### 11.1 内存 TTL 缓存

文件：`backend/app/services/cache.py`。

`TtlCache` 是进程内缓存，key 为：

```text
provider:date:from_station:to_station
```

### 11.2 SQLite OD 查询缓存

`SqliteOdCache` 通过环境变量启用：

```text
ROUTE_SEGMENT_CACHE_DB
ROUTE_SEGMENT_CACHE_TTL_SECONDS
```

用于缓存 provider 的 OD 查询结果，减少重复远程请求。命中后会回填内存缓存，并且不增加 `remote_query_count`。

### 11.3 SQLite 静态票价库

文件：`backend/app/services/static_price_repository.py`。

核心表：

| 表 | 职责 |
| --- | --- |
| `station_snapshot` | 预留保存站点名称、电报码、拼音和抓取时间等站点快照信息 |
| `od_public_price_snapshot` | 保存某日期、某 OD、某车次、某席别的公布票价 |
| `od_min_price_snapshot` | 保存某日期、某 OD 的最低价聚合结果 |
| `crawl_job` | 保存一次刷新任务的整体状态 |
| `crawl_task` | 保存刷新任务内每个 OD 的执行状态 |

说明：当前代码已经在 `SQLiteStaticPriceRepository._ensure_schema()` 中创建 `station_snapshot` 表；但静态票价刷新和查询主链路仍以 OD 票价快照为核心，站点快照的完整写入和使用闭环仍属于后续扩展。

## 12. 搜索遥测与静态票价闭环

遥测文件：`backend/app/services/search_telemetry.py`。

启用环境变量：`SEARCH_TELEMETRY_DB`。

表：

| 表 | 职责 |
| --- | --- |
| `search_runs` | 记录每次搜索的查询条件、方案数量、最佳价格、最佳中转站和远程请求数 |
| `transfer_station_hits` | 聚合某 OD 下中转站命中次数和历史最佳价格 |

闭环链路：

```text
用户搜索
  ↓
RouteSearchService._record_search()
  ↓
search_runs / transfer_station_hits
  ↓
backend/scripts/export_refresh_od_from_telemetry.py
  ↓
data/refresh_od.csv
  ↓
backend/scripts/refresh_static_prices.py
  ↓
data/static_prices.sqlite3
  ↓
LocalStaticPriceProvider
  ↓
RouteSearchService 后续搜索优先本地命中
```

### 12.1 遥测导出脚本

文件：`backend/scripts/export_refresh_od_from_telemetry.py`。

能力：从 `search_runs` 聚合高频查询 OD；从 `transfer_station_hits` 展开 `A -> T` 和 `T -> B`；合并重复 OD；输出 `origin,destination,priority,reason` CSV。

### 12.2 静态票价刷新脚本

文件：`backend/scripts/refresh_static_prices.py`。

能力：读取 OD CSV、调用远程 provider、限速、部分失败不中断整体任务、写入 `crawl_job` / `crawl_task`、写入静态票价库。

当前脚本读取 `origin/destination`，会忽略 `priority/reason` 额外列；`crawl_task.priority` 当前固定写入 `100`。按优先级刷新仍是后续增强。

### 12.3 静态票价库诊断脚本

文件：`backend/scripts/inspect_static_prices.py`。

能力：查看静态库整体覆盖；查询指定日期和 OD 的车次数、席别覆盖、最低价、最低价车次、最低价席别和更新时间。

  ↓

  ↓ 未命中

  ↓

  ↓

  ↓

```text
TRAIN_DATA_PROVIDER=static-price

  ↓

返回本次结果
```

## 17. 当前边界和后续演进

当前边界：

1. `12306-public-price` 依赖非公开稳定接口，存在失败、限流和字段变化风险。
2. 静态票价库表示公布票价或历史观测票价，不表示实时余票。
3. 当前二换搜索的启发函数为 `0`，等价 Uniform Cost Search。
4. 历史最低价不是安全 lower bound，不能用于硬剪枝。
5. SQLite 适合当前单机 MVP；多实例并发写或多人共享场景后再考虑 PostgreSQL。
6. 刷新脚本应作为独立脚本、定时任务或 worker 执行，不应放入 FastAPI 请求链路。
7. 前端当前固定查询一次中转，尚未暴露最大中转次数配置。

后续演进：

1. 增强 `refresh_static_prices.py`，读取并使用 `priority/reason`，按优先级刷新并输出失败报告。
2. 扩充 `data/seed_od.csv` 和由遥测导出的 `refresh_od.csv`，提高静态票价库覆盖率。
3. 前端更明确展示 `mock`、`12306-public-price`、`static-price` 的数据来源和更新时间。
4. 增加静态库覆盖率、搜索遥测、缓存命中和远程请求下降情况的诊断能力。
5. 建立更完整的站点和车次经停知识库。
6. 将刷新任务从手动脚本升级为定时任务或独立 worker。
7. 在能证明不高估的前提下设计 A* lower bound。
