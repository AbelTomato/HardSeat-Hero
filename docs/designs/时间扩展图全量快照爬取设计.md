# 时间扩展图全量快照爬取设计

## 1. 目标

本文档定义时间扩展图基础边数据的全量爬取方案。目标是通过长时间离线任务，获取指定采集日期的：

```text
当日车次清单
  ↓
单车全经停站序和到发时刻
  ↓
同车任意 OD 公布票价
  ↓
train_od_fare_edge 基础边
```

在线搜索阶段只读取本地 SQLite，不访问 12306 或其他远程接口。

## 2. 核心判断

当前采用直接全量暴力爬取策略。按用户给出的规模估算：

```text
每日车次数：约 10,000+
平均每车停站数：约 15
每车 OD 组合数：15 * 14 / 2 = 105
平均席别数：约 4
```

注意：席别数影响写入边数量，不应直接等同于票价接口请求数。若 `queryAllPublicPrice` 对一个 OD 返回该 OD 下多车、多席别价格，则请求量主要由“车次 OD 组合”决定，写入量再乘席别数。

粗略规模：

```text
票价请求量 ≈ 10,000 * 105 = 1,050,000 次
写入边数量 ≈ 1,050,000 * 4 = 4,200,000 条 seat edge
```

该规模可以通过长时间挂机、限速、并发、重试、断点续跑完成。设计重点不是避免暴力爬取，而是保证任务可恢复、可观察、可限速、可失败隔离。

## 3. 数据源接口

### 3.1 车次清单

```text
GET https://kyfw.12306.cn/otn/queryTrainInfo/getTrainName?date=YYYY-MM-DD
```

用途：获取指定日期可查询车次列表。

已验证事实：

- 响应字段包含 `station_train_code`、`train_no`。
- 响应存在重复 `train_no`，必须去重。
- 样例日期响应约 16,263 条原始记录，按 `train_no` 去重约 11,847 条。

### 3.2 单车经停时刻

```text
GET https://kyfw.12306.cn/otn/queryTrainInfo/query?leftTicketDTO.train_no=...&leftTicketDTO.train_date=YYYY-MM-DD&rand_code=
```

用途：获取单个 `train_no` 的经停站序和到发时刻。

核心字段：

| 字段                 | 含义                             |
| -------------------- | -------------------------------- |
| `station_name`       | 站名                             |
| `station_train_code` | 展示车次                         |
| `station_no`         | 站序，字符串如 `01`              |
| `arrive_time`        | 到达时刻，始发站可能为 `----`    |
| `start_time`         | 发车时刻，终到站可能等于到达时刻 |
| `arrive_day_diff`    | 到达日偏移                       |
| `running_time`       | 相对始发运行时长                 |
| `start_station_name` | 始发站                           |
| `end_station_name`   | 终到站                           |

### 3.3 公布票价

```text
GET https://kyfw.12306.cn/otn/leftTicketPrice/queryAllPublicPrice?leftTicketDTO.train_date=YYYY-MM-DD&leftTicketDTO.from_station=FROM&leftTicketDTO.to_station=TO&purpose_codes=ADULT
```

用途：获取某个 OD 的公布票价。

项目已有实现：

- `backend/app/adapters/railway_12306_public_price.py`
- `StationCodeRepository`
- `PublicPriceClient`
- `PRICE_FIELDS`
- `parse_price()`
- `parse_duration_minutes()`

全量爬虫应复用这些解析逻辑，不重复维护席别字段映射。

## 4. 全量爬取主流程

```text
1. 获取站名 -> 电报码映射
2. 请求 getTrainName(date)
3. 按 train_no 去重，保留 train_code
4. 初始化 train task
5. 并发限速请求每个 train_no 的 queryTrainInfo
6. 写入 train_service_snapshot / train_stop_snapshot
7. 对每个 stop list 生成所有 i < j 的 OD 组合
8. 初始化 price task
9. 并发限速请求每个 OD 的 queryAllPublicPrice
10. 从返回车次中过滤当前 train_no 或 train_code
11. 将席别价格展开为 TrainOdFareEdge
12. upsert 到 train_od_fare_edge
13. 记录任务状态和统计信息
```

## 5. OD 组合策略

默认对每车全量生成 `i < j`：

```python
for i in range(len(stops)):
    for j in range(i + 1, len(stops)):
        yield stops[i], stops[j]
```

默认不跳过短区间。短区间对时间扩展图换乘搜索有价值。

仍需提供控制参数，便于调试、灰度和异常情况下缩小任务规模：

| 参数                    | 默认 | 含义                                   |
| ----------------------- | ---: | -------------------------------------- |
| `--limit-trains`        |  `0` | 限制车次数，`0` 表示不限制             |
| `--max-stops-per-train` |  `0` | 单车最大停站数，`0` 表示不限制         |
| `--max-od-per-train`    |  `0` | 单车最大 OD 任务数，`0` 表示不限制     |
| `--min-station-gap`     |  `1` | 最小站序差，默认所有相邻及以上 OD      |
| `--include-seat-types`  |   空 | 只导入指定席别；空表示全部支持席别     |
| `--dry-run`             | 关闭 | 只生成任务和统计，不请求票价或写基础边 |

## 6. 并发、延时与重试

### 6.1 双队列限速

建议将时刻表请求和票价请求拆成两个独立队列：

| 队列           | 默认并发 | 默认请求间隔 | 说明                        |
| -------------- | -------: | -----------: | --------------------------- |
| `train-info`   |        4 |       0.3 秒 | 单车经停详情，数量约 1 万级 |
| `public-price` |        2 |       0.8 秒 | OD 票价请求，数量约百万级   |

CLI 参数：

```text
--train-info-concurrency 4
--price-concurrency 2
--train-info-delay 0.3
--price-delay 0.8
```

### 6.2 重试策略

默认最大重试 5 次：

```text
第 1 次失败：等待 2 秒
第 2 次失败：等待 4 秒
第 3 次失败：等待 8 秒
第 4 次失败：等待 16 秒
第 5 次失败：等待 32 秒
```

可加入 0 到 1 秒随机抖动，避免多个 worker 同时重试。

可重试错误：

- 网络超时。
- 连接失败。
- HTTP 5xx。
- 12306 返回非 JSON。
- 返回 HTML / 风控页。
- `status=false` 且错误信息不明确。

不可重试或低优先级重试错误：

- 站名无法映射电报码。
- `queryTrainInfo` 返回空停站。
- 票价接口正常返回但不包含目标车次。

其中“不包含目标车次”应记录为 `skipped`，不是全局失败。

## 7. 断点续跑与任务状态

### 7.1 车次任务表

```sql
CREATE TABLE IF NOT EXISTS crawl_full_snapshot_train_task (
    provider TEXT NOT NULL,
    crawl_date TEXT NOT NULL,
    train_no TEXT NOT NULL,
    train_code TEXT NOT NULL,
    status TEXT NOT NULL,
    attempts INTEGER NOT NULL DEFAULT 0,
    stop_count INTEGER NOT NULL DEFAULT 0,
    last_error TEXT,
    created_at TEXT NOT NULL,
    updated_at TEXT NOT NULL,
    PRIMARY KEY(provider, crawl_date, train_no)
);
```

### 7.2 票价任务表

```sql
CREATE TABLE IF NOT EXISTS crawl_full_snapshot_price_task (
    provider TEXT NOT NULL,
    crawl_date TEXT NOT NULL,
    train_no TEXT NOT NULL,
    train_code TEXT NOT NULL,
    from_station TEXT NOT NULL,
    to_station TEXT NOT NULL,
    from_station_no INTEGER NOT NULL,
    to_station_no INTEGER NOT NULL,
    status TEXT NOT NULL,
    attempts INTEGER NOT NULL DEFAULT 0,
    edge_count INTEGER NOT NULL DEFAULT 0,
    last_error TEXT,
    created_at TEXT NOT NULL,
    updated_at TEXT NOT NULL,
    PRIMARY KEY(provider, crawl_date, train_no, from_station_no, to_station_no)
);
```

### 7.3 状态流转

```text
pending -> running -> succeeded
pending -> running -> failed
pending -> running -> skipped
```

启动时处理：

- `--resume`：跳过 `succeeded`，继续 `pending/failed/running`。
- `--retry-failed`：将 `failed` 重置为 `pending`。
- 上次异常退出留下的 `running` 必须重置为 `pending`。

## 8. 数据写入

### 8.1 经停快照

`queryTrainInfo` 成功后写入已有表：

```text
train_service_snapshot
train_stop_snapshot
```

这些表仍然按 `service_date` 保存，用于追踪采集样本日的原始经停数据。

### 8.2 基础边

票价查询成功后写入：

```text
train_od_fare_edge
```

基础边不保存 `service_date`，只保存相对时刻和 day offset。在线搜索时再按用户查询日期展开。

字段映射：

| `train_od_fare_edge` 字段 | 来源                                                        |
| ------------------------- | ----------------------------------------------------------- |
| `provider`                | CLI 参数，默认 `full-snapshot`                              |
| `train_no`                | `getTrainName.train_no` / 票价 DTO `train_no`               |
| `train_code`              | `station_train_code` 去掉括号后的展示车次                   |
| `from_station`            | stop[i].station_name                                        |
| `to_station`              | stop[j].station_name                                        |
| `from_station_no`         | stop[i].station_no                                          |
| `to_station_no`           | stop[j].station_no                                          |
| `depart_time`             | stop[i].start_time                                          |
| `arrive_time`             | stop[j].arrive_time 或 stop[j].start_time                   |
| `depart_day_offset`       | stop[i] 的 day offset，第一版可按 running_time 推导或默认 0 |
| `arrive_day_offset`       | stop[j].arrive_day_diff                                     |
| `duration_minutes`        | 两站 running_time 差，或票价 DTO `lishi`                    |
| `seat_type`               | `PRICE_FIELDS` 映射                                         |
| `price`                   | 价格字段除以 10 后的 Decimal 字符串                         |
| `source`                  | `12306-full-snapshot`                                       |
| `fetched_at`              | 当前 UTC 时间                                               |
| `raw_hash`                | 原始 DTO JSON hash                                          |

## 11. 验收标准

- [ ] 能获取指定日期车次清单并去重。
- [ ] 能获取单车经停站序和到发时刻。
- [ ] 能为单车生成全部 `i < j` OD 票价任务。
- [ ] 能按 OD 查询公布票价并筛选目标车次。
- [ ] 能把席别价格写入 `train_od_fare_edge`。
- [ ] 重复运行不会重复插入，能够 upsert。
- [ ] 中断后 `--resume` 能继续未完成任务。
- [ ] 失败任务有错误记录，不阻塞其他任务。
- [ ] 在线搜索阶段不访问远程接口。

## 12. 风险

1. 12306 页面接口可能变更、限流或返回风控页。
2. 百万级请求耗时长，必须允许多次中断和恢复。
3. 采集日期只代表样本日，不等价于长期运营日历。
4. 某些 OD 票价接口可能不返回目标车次，需要记录 `skipped`。
5. SQLite 写入数可能达到数百万，需使用批量事务和索引控制写入成本。
6. 单车跨日运行的 `depart_day_offset` 需要谨慎推导，第一版可优先保证 `arrive_day_offset` 正确。
