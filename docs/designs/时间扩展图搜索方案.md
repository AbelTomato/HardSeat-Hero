# 时间扩展图搜索方案

## 背景

当前中转搜索依赖候选中转站、OD 查询预算和启发式扩展顺序。在 `static-with-remote-fallback` 模式下，已经观察到一个明确问题：

```text
厦门 -> 深圳，2026-07-12，最大中转 2 次，候选站上限 100
```

- `赣州` 是候选站第 1。
- `厦门 -> 赣州` 本地静态库有数据，最低 `62.5`。
- `赣州 -> 深圳` 本地静态库缺少 `2026-07-12/2026-07-13` 数据。
- 在 fallback 模式下，理论上应触发 `12306-public-price` 补查。
- 但当前二换 A* 搜索会先消耗大量 OD 查询预算，导致关键 OD `赣州 -> 深圳` 没有机会触发 fallback。

这说明当前问题不只是候选站召回，而是搜索模型仍以“按 OD 试探”为核心，难以保证全局最优，也难以控制远程 fallback 成本。

进一步看，`12306-public-price` 公布票价和 `queryTrainInfo` 时刻表都属于低频变化数据。把它们放进用户查询链路实时调用并不合理：

1. Dijkstra / A* 建图过程需要大量边，实时网络 I/O 会使搜索延迟不可控。
2. 远程接口可能限流、失败、变更字段或要求 cookie，不能作为在线最短路算法的组成部分。
3. 公布票价和时刻表相对稳定，更适合作为离线静态数据提前采集并持久化。

因此，时间扩展图方案应从“搜索时 fallback 补洞”调整为：

```text
离线采集并持久化时刻表与公布票价，在线阶段只访问本地数据库，基于本地静态数据构建时间扩展图并执行最短路搜索。
```

## 目标

长期目标是把路线搜索从“候选中转站试探”调整为：

```text
离线静态数据建库，在线本地时间扩展图最短路搜索。
```

核心目标：

1. 在线搜索阶段不访问 12306，不存在实时网络 I/O。
2. 降低对候选中转站的正确性依赖。
3. 避免搜索过程把预算浪费在大量无效 OD 上。
4. 支持跨日、多换乘、最短换乘时间、最大总耗时等约束。
5. 基于完整经停表和公布票价表，为同车合法 OD 建边。
6. 为后续增量刷新、数据版本和余票实时校验预留边界。

## 在线搜索硬约束

在线查询链路禁止调用以下远程接口：

```text
12306-public-price
queryTrainInfo/getTrainName
queryTrainInfo/query
search/v1/train/search
```

所有远程接口调用只能发生在：

1. 离线采集脚本。
2. 定时任务。
3. 独立 worker。
4. 手动数据刷新命令。

在线搜索只允许访问本地数据库或本地预构建图数据。如果本地数据缺失，搜索结果应明确标记为“静态库无覆盖”，而不是在 Dijkstra 过程中触发远程补查。

## 静态数据源

### 1. 车次时刻表数据

来源：`data/12306_query_train_info_probe` 已验证的 `queryTrainInfo` 页面接口链路。

采集链路：

```text
queryTrainInfo/getTrainName?date=yyyy-MM-dd
        ↓
获取指定日期车次清单，按 train_no 去重
        ↓
queryTrainInfo/query?leftTicketDTO.train_no=...&leftTicketDTO.train_date=yyyy-MM-dd
        ↓
获取单车完整停站时刻表
        ↓
写入 train_service_snapshot / train_stop_snapshot
```

已验证样例字段：

```text
station_train_code
train_no
station_name
start_time
arrive_time
station_no
running_time
start_station_name
end_station_name
arrive_day_diff
```

注意事项：

- `getTrainName` 响应含重复 `train_no`，必须去重。
- `arrive_time=----` 表示始发站无到达时间，需要归一化为空值或等于发车时间。
- 跨日必须通过 `service_date + HH:MM + arrive_day_diff` 转成绝对 datetime。
- 该数据只表示运行图和停站时刻，不表示余票和可售状态。

### 2. 公布票价数据

来源：现有 `12306-public-price` provider。

使用方式从“在线 fallback”调整为“离线批量采集”：

```text
基于 train_stop_snapshot 派生同车合法 OD
        ↓
批量调用 12306-public-price
        ↓
保存 train_od_price_snapshot / od_public_price_snapshot
        ↓
在线搜索直接读取本地票价
```

公布票价低频变化，第一阶段可先不设计复杂 TTL。是否需要更新、如何增量更新、是否按日期或车次刷新，后续单独设计。

### 3. 全量的边界

这里的“全量”不应理解成全国所有车站 `N²` 任意 OD 暴力枚举。更合理的范围是：

```text
全量采集指定日期的车次清单
全量采集这些车次的停站序列
基于每趟车的停站序列派生该车可售方向上的 OD
采集这些同车合法 OD 的公布票价
```

这样采集范围由真实车次和停站序列限定，比盲目枚举全国站点对更可控，也更适合后续建图。

## 数据模型

### `train_service_snapshot`

保存指定日期的车次级快照。

```sql
CREATE TABLE IF NOT EXISTS train_service_snapshot (
    provider TEXT NOT NULL,
    service_date TEXT NOT NULL,
    train_no TEXT NOT NULL,
    train_code TEXT NOT NULL,
    start_station TEXT NOT NULL,
    end_station TEXT NOT NULL,
    fetched_at TEXT NOT NULL,
    raw_hash TEXT,
    PRIMARY KEY(provider, service_date, train_no)
);

CREATE INDEX IF NOT EXISTS idx_train_service_code
ON train_service_snapshot(provider, service_date, train_code);
```

### `train_stop_snapshot`

保存每趟车的完整经停序列和到发时刻。

```sql
CREATE TABLE IF NOT EXISTS train_stop_snapshot (
    provider TEXT NOT NULL,
    service_date TEXT NOT NULL,
    train_no TEXT NOT NULL,
    train_code TEXT NOT NULL,
    station_name TEXT NOT NULL,
    station_no INTEGER NOT NULL,
    arrive_at TEXT,
    depart_at TEXT,
    arrive_day_diff INTEGER NOT NULL DEFAULT 0,
    running_time TEXT,
    fetched_at TEXT NOT NULL,
    raw_hash TEXT,
    PRIMARY KEY(provider, service_date, train_no, station_no)
);

CREATE INDEX IF NOT EXISTS idx_train_stop_station_time
ON train_stop_snapshot(provider, service_date, station_name, depart_at);

CREATE INDEX IF NOT EXISTS idx_train_stop_train
ON train_stop_snapshot(provider, service_date, train_no, station_no);
```

字段约束：

- `arrive_at` / `depart_at` 应保存绝对 datetime，而不是只保存 `HH:MM`。
- 始发站 `arrive_at` 可为空。
- 终到站 `depart_at` 可等于 `arrive_at` 或为空，但建图时不能从终到站继续生成同车出发边。

### `train_od_price_snapshot`

保存同车合法上下车区间的公布票价。

```sql
CREATE TABLE IF NOT EXISTS train_od_price_snapshot (
    provider TEXT NOT NULL,
    service_date TEXT NOT NULL,
    train_no TEXT NOT NULL,
    train_code TEXT NOT NULL,
    from_station TEXT NOT NULL,
    to_station TEXT NOT NULL,
    from_station_no INTEGER NOT NULL,
    to_station_no INTEGER NOT NULL,
    depart_at TEXT NOT NULL,
    arrive_at TEXT NOT NULL,
    duration_minutes INTEGER NOT NULL,
    seat_type TEXT NOT NULL,
    price TEXT NOT NULL,
    currency TEXT NOT NULL DEFAULT 'CNY',
    source TEXT NOT NULL,
    fetched_at TEXT NOT NULL,
    raw_hash TEXT,
    PRIMARY KEY(
        provider,
        service_date,
        train_code,
        from_station,
        to_station,
        seat_type
    )
);

CREATE INDEX IF NOT EXISTS idx_train_od_price_query
ON train_od_price_snapshot(provider, service_date, from_station, to_station, price);

CREATE INDEX IF NOT EXISTS idx_train_od_price_train
ON train_od_price_snapshot(provider, service_date, train_no, from_station_no, to_station_no);
```

说明：

- 同车任意 OD 票价不一定等于相邻区间票价之和，不能把一条长 OD 票价随意拆成相邻边再累加。
- 有 `from_station -> to_station` 的同车 OD 票价，才为该区间建乘车边。
- 如果 `12306-public-price` 只能按 OD 返回多车次结果，则离线任务以 OD 为采集单位，入库时再按 `train_code` / `train_no` 关联。

### 与现有 `od_public_price_snapshot` 的关系

现有 `od_public_price_snapshot` 本质是 OD 票价快照：

```text
provider, travel_date, origin, destination, train_no, depart_at, arrive_at, seat_type, price
```

它可以继续保留，用于：

1. 兼容已有 `LocalStaticPriceProvider`。
2. 快速 OD 查询。
3. 直达最低价展示。
4. 从离线票价采集结果同步派生。

但时间扩展图的主数据源应逐步切换到：

```text
train_stop_snapshot + train_od_price_snapshot
```

## 基本模型

采用时间扩展图（time-expanded graph）。

### 节点

节点表示车站上的时间事件：

```python
EventNode:
    station: str
    time: datetime
    kind: Literal["arrival", "departure"]
    train_no: str | None
```

查询时额外加入虚拟节点：

```text
source
sink
```

### 边

#### 1. 乘车边

表示乘坐某趟车从一个站到另一个站：

```text
departure_event(A, t1, train=T) -> arrival_event(B, t2, train=T)
```

权重：

```text
票价
```

元数据：

```text
train_no, train_code, from_station, to_station, depart_at, arrive_at, seat prices
```

建边来源：

```text
train_stop_snapshot 提供同车时刻和站序
train_od_price_snapshot 提供 from_station -> to_station 的公布票价
```

#### 2. 换乘边

同一车站内，某个到达事件可以连接到满足最短换乘时间的第一个出发事件：

```text
arrival_event(station, 14:00) -> departure_event(station, 14:20)
```

条件：

```text
departure_time >= arrival_time + min_transfer_minutes
```

权重：

```text
0
```

#### 3. 等待边

同一车站内按时间排序，把事件线性连接：

```text
event_i -> event_i+1
```

权重：

```text
0
```

这样可以避免为每个到达事件连接所有后续出发事件，复杂度从潜在的 `O(A * D)` 降为 `O(E_station)`。

## 搜索算法

第一阶段可使用 Dijkstra。

基础权重：

```text
total_price
```

状态至少需要包含：

```python
SearchLabel:
    node_id: int
    total_price: Decimal
    transfer_count: int
    first_depart_at: datetime | None
    path_edges: tuple[int, ...]
```

约束：

```text
transfer_count <= query.max_transfers
total_duration <= query.max_total_duration_minutes（如果设置）
departure_time >= arrival_time + query.min_transfer_minutes
```

建图日期范围建议：

```text
query.date
query.date + 1
query.date + 2
```

如果后续要同时优化价格、到达时间、换乘舒适度，应升级为 label-setting / Pareto 最短路，而不是单标签 Dijkstra。

## 离线采集链路

### 阶段一：采集时刻表

目标：建立可用于建图的完整停站数据。

流程：

```text
输入 service_date
        ↓
调用 getTrainName(service_date)
        ↓
按 train_no 去重
        ↓
逐 train_no 调 query(train_no, service_date)
        ↓
归一化站序、到发时间、跨日信息
        ↓
写入 train_service_snapshot / train_stop_snapshot
```

测试重点：

- 重复 `train_no` 去重。
- 始发站、终到站特殊时间处理。
- `arrive_day_diff` 跨日归一化。
- 同一车次重复采集可幂等覆盖。

### 阶段二：采集公布票价

目标：建立本地可直接建边的同车 OD 票价数据。

流程：

```text
读取 train_stop_snapshot
        ↓
按每趟车 station_no 派生 from_station_no < to_station_no 的合法 OD
        ↓
合并相同 from_station -> to_station -> service_date，减少重复远程请求
        ↓
调用 12306-public-price
        ↓
按 train_code / train_no 匹配目标车次
        ↓
写入 train_od_price_snapshot
        ↓
可选同步派生 od_public_price_snapshot / od_min_price_snapshot
```

失败处理：

- 写入 `crawl_task`。
- 记录错误和重试次数。
- 后续离线重试。
- 不影响在线搜索链路。

### 阶段三：在线本地图搜索

目标：用户查询时只读取本地静态数据。

流程：

```text
接收 origin, destination, date, max_transfers, min_transfer_minutes
        ↓
读取 date ~ date+2 的 train_stop_snapshot 和 train_od_price_snapshot
        ↓
构建时间扩展图或读取预构建边表
        ↓
执行 Dijkstra / label-setting
        ↓
返回最低价路径、换乘信息、数据更新时间
```

无数据时的行为：

```text
返回“静态库无覆盖 / 当前本地数据不足”，不进行远程 fallback。
```

### 阶段四：刷新策略

第一阶段不需要复杂实时更新策略。推荐先支持：

1. 手动全量刷新指定日期。
2. 定时刷新未来若干天。
3. 失败任务重试。
4. 采集结果 `raw_hash` 对比，用于后续判断数据是否变化。

是否需要更新、如何增量更新、保留多少历史版本，后续根据数据规模和接口稳定性再设计。

## 与现有实现的关系

短期不建议直接删除 `CandidateTransferStationGenerator`。它仍可用于：

1. 离线票价采集优先级排序。
2. 数据覆盖不足时生成高价值补采任务。
3. UI 搜索范围提示。
4. 旧搜索链路的降级策略。

但路线搜索正确性应逐步迁移到本地时序图，而不是依赖候选站枚举。

`static-with-remote-fallback` 也应调整定位：

```text
它可以作为旧 OD 搜索链路的临时模式，但不应作为时间扩展图搜索的在线模式。
```

时间扩展图搜索只接受本地静态数据输入。

## 待验证事项

后续测试新数据源和采集脚本时，重点验证：

1. `getTrainName(date)` 在目标日期范围内是否稳定返回完整车次清单。
2. `query(train_no, date)` 是否能稳定返回完整经停、站序和跨日字段。
3. `12306-public-price` 是否能覆盖同车任意合法 OD 的公布票价。
4. 票价接口返回结果能否可靠按 `train_code` / `train_no` 关联到时刻表车次。
5. 全量采集一天的请求量、耗时、失败率和限流风险。
6. SQLite 在目标数据量下的写入和查询性能。
7. `arrive_at` / `depart_at` 绝对时间归一化是否正确处理跨日。
8. 建图时是否严格禁止把长 OD 票价拆成相邻区间票价相加。
9. 在线查询链路是否完全无远程网络 I/O。
10. 返回结果是否展示静态数据更新时间和“不保证实时有票”的语义。
