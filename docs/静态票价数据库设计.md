# 静态票价数据库设计

## 目标

当前已经建立本地 SQLite 静态公布票价库，使路线搜索在已有数据范围内无需访问 12306，即可完成直达和中转最低价查询。本文描述当前已落地表结构和保留的后续扩展项。

当前架构：

```text
定时全量/准全量采集公布票价
        ↓
写入本地 SQLite 静态查询库
        ↓
用户查询时优先查本地库
        ↓
缺失/过期/异常 OD 再按配置调用远程 Provider
        ↓
远程结果反写为观测数据或静态快照
```

本阶段重点不是继续微调搜索循环，而是减少真实数据源远程 I/O。铁路公布票价相对稳定，适合本地持久化和静态查询。

## 产品语义

静态票价库提供的是：

```text
公布票价 / 历史观测最低票价路径
```

不提供：

```text
实时余票保证
实时停运保证
动态折扣保证
```

因此前端或接口应明确展示数据更新时间，例如：

```text
价格数据更新时间：2026-06-26 08:00，本结果不保证实时有票。
```

如果未来增加余票能力，应作为独立实时校验层，不应和静态公布票价库混用同一 TTL 逻辑。

## 核心约束

1. 不应直接枚举全国所有 OD 暴力爬取。
2. 历史最低价可以用于候选排序、刷新优先级和展示，但不能作为 A* 硬剪枝 lower bound。
3. API 服务进程不负责长周期爬虫任务，爬虫应通过脚本、定时任务或独立 worker 执行。
4. `TrainDataProvider` 边界保持不变，静态库作为新的 provider 或 provider 包装层接入。
5. SQLite 第一阶段足够，表结构应避免绑定 SQLite 专属能力，保留迁移 PostgreSQL 的空间。

## 推荐数据库选择

第一阶段使用 SQLite。

理由：

- 项目当前为单机 MVP / 本地服务形态。
- 已存在 SQLite OD 缓存和搜索遥测思路，技术栈一致。
- 静态票价库读多写少，适合 SQLite。
- 零运维，便于本地演示和测试。

后续当出现多进程并发写、中心化服务、多人共享数据时，再迁移 PostgreSQL。

## 表结构

### `station_snapshot`（后续扩展，当前未落库）

保存车站快照，用于站名、电报码和站点搜索。当前 `SQLiteStaticPriceRepository` 尚未创建该表，站点能力仍由 12306 站点 JS 解析和候选站元数据承担。

```sql
CREATE TABLE IF NOT EXISTS station_snapshot (
    provider TEXT NOT NULL,
    station_name TEXT NOT NULL,
    station_code TEXT NOT NULL,
    pinyin TEXT,
    short_pinyin TEXT,
    is_active INTEGER NOT NULL DEFAULT 1,
    fetched_at TEXT NOT NULL,
    PRIMARY KEY(provider, station_name)
);

CREATE INDEX IF NOT EXISTS idx_station_snapshot_code
ON station_snapshot(provider, station_code);
```

用途：

- 替代每次动态下载站点 JS。
- 支持前端站点自动补全。
- 为后续采集任务提供站点范围。

### `od_public_price_snapshot`

保存某日期、某 OD、某车次、某席别的公布票价。该表是静态查询主表。

```sql
CREATE TABLE IF NOT EXISTS od_public_price_snapshot (
    provider TEXT NOT NULL,
    travel_date TEXT NOT NULL,
    origin TEXT NOT NULL,
    destination TEXT NOT NULL,
    train_no TEXT NOT NULL,
    train_code TEXT NOT NULL,
    depart_at TEXT NOT NULL,
    arrive_at TEXT NOT NULL,
    duration_minutes INTEGER NOT NULL,
    seat_type TEXT NOT NULL,
    price TEXT NOT NULL,
    currency TEXT NOT NULL DEFAULT 'CNY',
    source TEXT NOT NULL,
    fetched_at TEXT NOT NULL,
    raw_hash TEXT,
    PRIMARY KEY(
        provider,
        travel_date,
        origin,
        destination,
        train_code,
        seat_type
    )
);

CREATE INDEX IF NOT EXISTS idx_od_price_query
ON od_public_price_snapshot(provider, travel_date, origin, destination, price);

CREATE INDEX IF NOT EXISTS idx_od_price_train
ON od_public_price_snapshot(provider, travel_date, train_code);
```

设计说明：

- `travel_date` 必须保留，运行图、停站和价格都可能随日期变化。
- `seat_type` 拆行存储，便于最低价查询和索引，不建议把全部席别塞入 JSON。
- `price` 使用 `TEXT` 存储 Decimal，避免浮点误差。
- `source` 记录数据来源，例如 `queryAllPublicPrice`、`manual`、`fixture`。
- `raw_hash` 用于判断同一 OD 响应是否发生变化，第一版可为空。

### `od_min_price_snapshot`

保存 OD 最低价聚合结果，用于快速查询、候选中转排序和展示更新时间。

```sql
CREATE TABLE IF NOT EXISTS od_min_price_snapshot (
    provider TEXT NOT NULL,
    travel_date TEXT NOT NULL,
    origin TEXT NOT NULL,
    destination TEXT NOT NULL,
    min_price TEXT NOT NULL,
    train_code TEXT NOT NULL,
    seat_type TEXT NOT NULL,
    depart_at TEXT NOT NULL,
    arrive_at TEXT NOT NULL,
    duration_minutes INTEGER NOT NULL,
    fetched_at TEXT NOT NULL,
    updated_at TEXT NOT NULL,
    PRIMARY KEY(provider, travel_date, origin, destination)
);

CREATE INDEX IF NOT EXISTS idx_od_min_price_origin_dest
ON od_min_price_snapshot(provider, origin, destination, min_price);
```

该表可由 `od_public_price_snapshot` 派生，也可在写入 OD 票价快照时同步维护。

用途：

- 快速判断某 OD 的历史最低公布票价。
- 候选中转站排序：优先尝试 `A -> T + T -> B` 历史价格低的中转站。
- 给用户展示价格数据更新时间。

### `crawl_job`

记录一次采集任务的整体状态。

```sql
CREATE TABLE IF NOT EXISTS crawl_job (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    provider TEXT NOT NULL,
    job_type TEXT NOT NULL,
    travel_date TEXT,
    scope TEXT NOT NULL,
    status TEXT NOT NULL,
    total_count INTEGER NOT NULL DEFAULT 0,
    success_count INTEGER NOT NULL DEFAULT 0,
    failed_count INTEGER NOT NULL DEFAULT 0,
    started_at TEXT,
    finished_at TEXT,
    error_message TEXT
);
```

建议 `job_type`：

```text
station_snapshot
popular_od_refresh
candidate_od_refresh
full_seed_refresh
```

### `crawl_task`

记录具体 OD 采集任务，支持限速、重试和中断恢复。

```sql
CREATE TABLE IF NOT EXISTS crawl_task (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    job_id INTEGER NOT NULL,
    provider TEXT NOT NULL,
    travel_date TEXT NOT NULL,
    origin TEXT NOT NULL,
    destination TEXT NOT NULL,
    priority INTEGER NOT NULL DEFAULT 100,
    status TEXT NOT NULL DEFAULT 'pending',
    attempts INTEGER NOT NULL DEFAULT 0,
    last_error TEXT,
    next_retry_at TEXT,
    created_at TEXT NOT NULL,
    updated_at TEXT NOT NULL,
    UNIQUE(job_id, travel_date, origin, destination)
);

CREATE INDEX IF NOT EXISTS idx_crawl_task_pick
ON crawl_task(status, priority, next_retry_at, id);
```

## 数据采集策略

### 不做无边界全国 OD 暴力爬取

全国车站 OD 是 `N²` 规模。直接枚举所有 `from -> to` 会导致：

- 请求量过大。
- 大量无效 OD。
- 容易触发远程限流。
- 爬取周期不可控。
- 数据库膨胀但查询命中率不一定高。

建议采用分层采集。

### 第一层：热门 OD 和用户查询 OD

来源：

- `search_telemetry`。
- `transfer_station_hits`。
- 一线城市、铁路枢纽、常见通勤和长途线路。

特点：

- 请求量小。
- 命中率最高。
- 对用户体验改善最直接。

### 第二层：候选中转 OD

对用户常查的 `A -> B`，扩展：

```text
A -> T
T -> B
```

`T` 来源：

- `CandidateTransferStationGenerator`。
- 静态低价断点表。
- 历史命中中转站。
- 铁路枢纽表。

这比全站 OD 更符合“最低价中转搜索”的产品目标。

### 第三层：线路/车次级别准全量

当具备完整经停表后，再考虑：

```text
车次 -> 经停站列表 -> 派生同车次可售 OD -> 查询/保存公布票价
```

这比盲目枚举全国 OD 更可控。

## 刷新周期建议

| 数据              |  建议周期 |
| ----------------- | --------: |
| 车站列表          |      7 天 |
| 热门 OD 票价      |      1 天 |
| 用户最近查询 OD   | 6~12 小时 |
| 候选中转 OD       |    1~3 天 |
| 大范围低优先级 OD |   7~30 天 |

刷新优先级应由用户查询频率、历史命中率、历史低价概率和数据年龄共同决定。

## 查询链路

当前已新增本地静态 provider 和远程补洞 wrapper：

```text
TrainDataProvider
    ├── LocalStaticPriceProvider
    │       └── SQLiteStaticPriceRepository
    └── StaticPriceFallbackProvider
            ├── LocalStaticPriceProvider
            ├── Railway12306PublicPriceProvider
            └── SQLiteStaticPriceRepository
```

`RouteSearchService._search_segments(A, B, date)` 查询顺序建议为：

```text
1. 内存 TTL 缓存
2. SQLite OD 车次缓存
3. SQLite 静态票价库
4. 可选远程 fallback provider
5. 远程结果反写静态票价库或观测库
```

当前支持两种模式：

| 模式                          | 行为                                            |
| ----------------------------- | ----------------------------------------------- |
| `static-only`                 | 只查本地静态库，速度快，适合离线演示和稳定查询  |
| `static-with-remote-fallback` | 本地无数据或过期时查远程 provider，并反写数据库 |

`static-with-remote-fallback` 的实际行为：

```text
LocalStaticPriceProvider 查询 SQLite 静态库
        ↓
命中且未过期：直接返回
        ↓
未命中或超过 STATIC_PRICE_MAX_AGE_DAYS：调用 fallback provider
        ↓
fallback 返回有价车次：写回 od_public_price_snapshot 和 od_min_price_snapshot
        ↓
返回本次远程结果

### 任务 4A：静态库诊断脚本

新增：

```text
backend/scripts/inspect_static_prices.py
```

能力：

- 查看静态库整体 OD 覆盖。
- 查看指定日期和 OD 的最低价、车次数、席别覆盖和更新时间。

### 任务 4B：从搜索遥测导出刷新 OD

新增：

```text
backend/scripts/export_refresh_od_from_telemetry.py
```

能力：

- 从 `search_runs` 导出高频查询 OD。
- 从 `transfer_station_hits` 展开 `A -> T` 和 `T -> B`。
- 输出 `origin,destination,priority,reason` CSV，供 `refresh_static_prices.py` 使用。
```

注意：该模式会在用户查询缺失 OD 时访问远程 12306 provider，查询耗时和失败率会受外部网络影响。

环境变量建议：

```powershell
TRAIN_DATA_PROVIDER=static-price
STATIC_PRICE_DB=D:\AbelTomato_Files\Developer\Projects\HardSeat-Hero\data\static_prices.sqlite3
STATIC_PRICE_MODE=static-with-remote-fallback
STATIC_PRICE_FALLBACK_PROVIDER=12306-public-price
STATIC_PRICE_MAX_AGE_DAYS=30
```

如果不设置 `STATIC_PRICE_MODE`，默认行为是 `static-only`。

`STATIC_PRICE_DB` 建议使用绝对路径。若使用相对路径，从项目根目录启动和从 `backend/` 目录启动会解析到不同位置，例如 `data/static_prices.sqlite3` 与 `backend/data/static_prices.sqlite3`。

## 与搜索算法的关系

静态票价库接入后：

1. `static-only` 下，本地已有 OD 直接返回，不产生远程请求。
2. `static-with-remote-fallback` 下，本地缺失或过期 OD 会按搜索过程触发远程补洞并写回静态库。
3. 历史最低价用于排序和刷新优先级。
4. A* 继续保持 `h=0`，或只使用能证明不高估的 lower bound。

禁止把历史最低价直接用于硬剪枝：

```text
历史最低价可能高于当前真实最低价，因此不是安全 admissible lower bound。
```

允许用途：

- 候选 OD 排序。
- 候选中转站排序。
- 爬虫任务优先级。
- 展示价格新鲜度。
- 判断是否需要远程刷新。

当前不建议把定时爬虫塞进 FastAPI 进程。

## 验收标准

第一阶段完成标准：

- `static-price` provider 可独立工作。
- 本地已有 OD 查询不产生远程请求。
- 本地缺失时可配置是否 fallback 到远程 provider。
- 查询结果仍走现有 `RouteSearchService`，不绕过搜索算法。
- 票价记录包含 `fetched_at` 或 `updated_at`。
- 数据过期策略可配置。
- 单元测试覆盖 repository、provider factory、搜索服务集成。
- 未启用静态库时现有 mock 和 12306 provider 行为不变。

## 不建议现在做

1. 不建议直接全站 OD 暴力爬取。
2. 不建议把爬虫放进 API 请求链路。
3. 不建议把历史价当 A* 硬剪枝下界。
4. 不建议马上引入 Redis/PostgreSQL。
5. 不建议混合余票实时状态和公布票价静态库。

## 结论

下一步应优先建设静态票价数据库，而不是继续细调本地搜索循环。

推荐短期目标：

```text
建立 SQLite 静态公布票价库，使路线搜索在已有数据范围内无需访问 12306，远程 Provider 只作为刷新和缺失补洞手段。
```

这条路线能直接降低真实数据源 I/O，提升稳定性，并为候选中转排序、历史低价展示和后续准全量采集打基础。
