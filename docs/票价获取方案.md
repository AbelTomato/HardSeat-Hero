# 票价获取方案

## 结论

`Parse12306` 和 `China-rail-way-stations-data` 只能提供车站、车次和时刻表数据，不能直接提供票价。

当前已验证可行的票价主路径不是旧的 `leftTicket/queryZ` + `queryTicketPrice`，而是 12306 官方公布票价页面使用的接口：

```text
GET https://kyfw.12306.cn/otn/leftTicketPrice/queryAllPublicPrice
```

该接口可直接按日期、出发站电报码、到达站电报码返回车次、时刻和公布票价。当前验证中，高铁动车和 K/Z 普速车均可返回价格字段。

## 推荐链路

1. 用 `station_name.js` 或 `station_name_new.js` 建立站名和电报码映射。
2. 调用 `leftTicketPrice/queryAllPublicPrice` 查询某日某 OD 的公布票价。
3. 从返回的 `queryLeftNewDTO` 中提取车次、区间、时间、历时和票价字段。
4. 将结果归一化为项目内 `TrainSegment` 和 `SeatPrice`。

该方案依赖 12306 非正式但官方页面使用的接口，不能作为强 SLA 数据源。实现时必须保留 Mock 数据源、缓存和失败降级。

## 接口链路

### 官方查询页面

```text
GET https://kyfw.12306.cn/otn/view/queryPublicIndex.html
```

页面标题为 `中国铁路12306网站`。静态分析页面后可见票价结果模板使用：

```text
queryLeftNewDTO.seatList[j].name
queryLeftNewDTO.seatList[j].price
```

页面主脚本：

```text
https://kyfw.12306.cn/otn/personalJS/dist/queryPublicIndex/main_v11106.js
```

该脚本中定义：

```javascript
queryAllPublicPrice = static_url_path + "/leftTicketPrice/queryAllPublicPrice"
```

实际调用方式为 `GET`。

### 车站电报码

优先使用：

```text
GET https://kyfw.12306.cn/otn/resources/js/framework/station_name.js
```

可选备用：

```text
GET https://kyfw.12306.cn/otn/personalJS/core/common/station_name_new.js
```

返回内容形如：

```text
@bjb|北京北|VAP|beijingbei|bjb|0
```

字段含义：

- 站名：第 2 项。
- 电报码：第 3 项。

### 公布票价查询

```text
GET https://kyfw.12306.cn/otn/leftTicketPrice/queryAllPublicPrice?leftTicketDTO.train_date=YYYY-MM-DD&leftTicketDTO.from_station=VNP&leftTicketDTO.to_station=AOH&purpose_codes=ADULT
```

建议请求头：

```text
Referer: https://kyfw.12306.cn/otn/view/queryPublicIndex.html
User-Agent: Mozilla/5.0 ...
Accept: application/json,text/javascript,*/*;q=0.01
```

已验证示例：

```text
北京南(VNP) -> 上海虹桥(AOH)
status=True
httpstatus=200
count=52
```

第一条样例：

```text
station_train_code: G547
train_no: 240000G54700
from_station_name: 北京南
to_station_name: 上海虹桥
start_time: 06:18
arrive_time: 12:11
lishi: 05:53
swz_price: 27820
zy_price: 12720
ze_price: 07950
```

## 返回字段

每条结果位于 `data[].queryLeftNewDTO`。

核心字段：

| 字段                    | 含义                                        |
| ----------------------- | ------------------------------------------- |
| `train_no`              | 内部车次编号                                |
| `station_train_code`    | 展示车次，如 `G547`、`K599`                 |
| `train_class_name`      | 列车类型，如 `高速`、`动车`、`快速`、`直特` |
| `from_station_telecode` | 查询区间出发站电报码                        |
| `from_station_name`     | 查询区间出发站名                            |
| `to_station_telecode`   | 查询区间到达站电报码                        |
| `to_station_name`       | 查询区间到达站名                            |
| `start_time`            | 出发时间                                    |
| `arrive_time`           | 到达时间                                    |
| `lishi`                 | 历时，如 `05:53`                            |
| `day_difference`        | 跨日天数                                    |
| `infoAll_list`          | 席别价格组合编码，可用于后续扩展解析        |

明确价格字段：

| 字段        | 席别     |
| ----------- | -------- |
| `swz_price` | 商务座   |
| `zy_price`  | 一等座   |
| `ze_price`  | 二等座   |
| `yz_price`  | 硬座     |
| `yw_price`  | 硬卧     |
| `rw_price`  | 软卧     |
| `gr_price`  | 高级软卧 |

价格编码以角为单位，解析为元时除以 10：

```text
07950 -> 795.0
12720 -> 1272.0
02510 -> 251.0
04560 -> 456.0
```

后端应使用 `Decimal` 解析，避免浮点误差。

## 与现有架构的关系

新增真实数据源适配器时保持现有边界：

```text
RouteSearchService
  -> TrainDataProvider
       -> MockTrainDataProvider
       -> Railway12306PublicPriceProvider
            -> StationCodeRepository
            -> PublicPriceClient
```

`RouteSearchService` 不直接依赖 12306 字段、URL 或请求头。

## 旧方案状态

旧链路：

```text
leftTicket/queryZ -> queryTicketPrice
```

当前环境下直接请求 `leftTicket/queryZ`、`leftTicket/query`、`leftTicket/queryA` 均返回 HTML 页面，标题为 `铁路客户服务中心`，不是 JSON。因此该链路不再作为主方案，仅保留为历史调研记录。

## 缓存策略

- 车站电报码：长缓存，建议 1 天到 7 天。
- 公布票价查询：短缓存，建议 5 到 15 分钟。
- 缓存 key：`12306-public-price:date:from_station:to_station`。
- 中转搜索会查询多组 OD，必须通过现有 `TtlCache` 或后续 Redis 限制重复请求。
- 持久化缓存粒度应与接口粒度一致，缓存对象是某日某 OD 的完整归一化结果，不是单个车次。
- 真实中转搜索必须设置单次请求预算；预算耗尽后只允许使用本地缓存，不继续请求 12306。

建议 SQLite 表：

```text
query_cache(
  provider,
  travel_date,
  origin,
  destination,
  fetched_at,
  expires_at,
  response_json
)
```

后续可在本地继续维护站点和线路知识库，用于候选中转站生成：

```text
station(name, telecode, lat, lon, centrality_score)
train_stop(train_no, station, stop_sequence, arrive_time, depart_time)
edge(from_station, to_station, train_no, duration_minutes, historical_lowest_price)
```

中转候选生成优先使用本地知识库，在线接口只负责刷新少量候选 OD 的公布票价。

## 失败降级

必须区分以下失败：

- 站名无法映射电报码。
- 12306 返回非 JSON、空结果或风控页。
- `data` 不是列表或缺少 `queryLeftNewDTO`。
- 票价字段缺失或为空。
- 网络超时或 TLS 失败。

降级策略：

- 真实数据源失败时不影响 Mock provider。
- 单个车次票价缺失时，跳过缺失席别；若无任何价格则不参与最低价排序。
- API 响应必须保留 `source` 和 `updated_at`，便于前端展示数据来源。

## 已知风险

- 12306 接口不是公开稳定 API，路径、字段、header 要求可能变化。
- 查询真实票价涉及实时服务和使用条款，产品化前需要确认合规边界。
- 查询一次中转会放大请求量，不能无缓存地逐站枚举。
- 地理直线剪枝不完全符合真实铁路拓扑，可能漏掉绕行但便宜的线路；后续需要引入铁路拓扑距离降低漏召回风险。
- 价格变化快，结果只能作为参考。
- `infoAll_list` 可支持更复杂席别，但第一版只解析明确 `*_price` 字段。
