# 总体设计

## 技术栈

- 前端：React、TypeScript、Vite、Tailwind CSS、本地 shadcn/ui 风格组件。
- 后端：FastAPI、Pydantic v2、httpx、pytest、SQLite。
- 数据源：Mock 数据源、12306 公布票价接口适配器、SQLite 静态票价库 provider、静态库远程 fallback provider。
- 缓存：内存 TTL 缓存；可选 SQLite OD 持久化缓存。Redis 仍属于未来扩展，不是当前实现。

## 后端模块

### domain

定义稳定领域模型：

- `StationMetadata`：候选中转站元数据。
- `TrainSegment`：单段车次。
- `SeatPrice`：席别、价格和余票状态占位。
- `RouteQuery`：查询条件。
- `TransferPlan`：中转方案。

### adapters

定义统一数据源接口：

```python
class TrainDataProvider:
    name: str

    async def search_segments(
        self,
        from_station: str,
        to_station: str,
        query: RouteQuery,
    ) -> list[TrainSegment]:
        ...

    def candidate_transfer_stations(self, query: RouteQuery) -> list[str]:
        ...
```

具体适配器：

- `MockTrainDataProvider`：本地样例数据，用于开发、测试和无网络演示。
- `Railway12306PublicPriceProvider`：封装 12306 公布票价查询链路。
- `LocalStaticPriceProvider`：读取 SQLite 静态票价库。
- `StaticPriceFallbackProvider`：静态库缺失或过期时调用远程 provider 并写回静态库。

### services

- `RouteSearchService`：组合直达、一换和二换方案，并维护搜索诊断。
- `TtlCache` / `SqliteOdCache`：内存与 SQLite OD 查询缓存。
- `SQLiteStaticPriceRepository`：静态公布票价库读写和最低价聚合。
- `SqliteSearchTelemetryRecorder`：搜索遥测和中转站命中记录。
- `CandidateTransferStationGenerator`：候选中转站生成。

## API 设计

`GET /api/health`

### 站点搜索

`GET /api/stations/search?q=北京`

### 方案查询

`POST /api/routes/search`

主前端页面使用流式接口：`POST /api/routes/search/stream`。该接口返回 NDJSON 事件流：搜索过程中持续返回 `type=snapshot` 快照事件，快照内包含当前已发现最低价 Top N 方案、`searched_count`、`pending_count`、`remote_query_count` 和 `elapsed_ms`；前端收到快照后整体替换结果列表，因此被更低价方案挤出的旧方案会自动消失。流结束前返回 `type=metadata` 元数据事件。一次性接口仍保留，用于测试和非流式调用，响应同样包含 `elapsed_ms`。

请求示例：

```json
{
  "from_station": "北京",
  "to_station": "上海",
  "date": "2026-07-01",
  "max_transfers": 1,
  "min_transfer_minutes": 30
}
```

返回示例：

```json
{
  "query_id": "20260701-bj-sh",
  "source": "mock",
  "updated_at": "2026-06-25T09:00:00+08:00",
  "elapsed_ms": 123,
  "plans": [
    {
      "total_price": 388.5,
      "total_duration_minutes": 520,
      "transfer_stations": ["南京南"],
      "segments": []
    }
  ]
}
```

## 算法概要

当前 MVP 目标固定为：查询最低价火车直达或中转方案。总耗时默认不再作为硬过滤条件；调用方只有显式传入 `max_total_duration_minutes` 时才启用方案总耗时过滤。换乘次数、最短换乘时间仍作为过滤条件，换乘等待舒适度和总耗时只作为同价辅助排序，不作为主优化目标。

### 当前实现：受控中转搜索

前端当前提交 `max_transfers=1`，不提交 `max_total_duration_minutes`，因此默认不限制方案总耗时。后端模型允许 `max_transfers=0..2`，`max_total_duration_minutes` 可选且最小值为 60 分钟；当该字段为 `null` 或缺省时不按总耗时过滤。当 `max_transfers >= 2` 时，后端启用 `CheapestPathSearcher`。该搜索器当前启发函数返回 `0`，等价 Uniform Cost Search，不使用历史最低价做硬剪枝。

一次中转链路：

1. 查询直达车次，得到直达方案和当前最低总价上界 `best_price`。
2. 生成候选中转站，避免枚举全国全量站点。
3. 查询 `出发站 -> 中转站` 和 `中转站 -> 目的站` 两段 OD 车次。
4. 过滤不满足换乘时间、时间顺序，以及显式总耗时约束的组合。
5. 计算每个方案总价、总耗时、换乘等待时间。
6. 按最低总价输出前 N 个方案，总耗时和换乘次数仅作为同价辅助排序。

### 真实数据源中转搜索约束

12306 公布票价接口的请求粒度是 `date + origin + destination` 的 OD 查询。中转搜索会把一次用户查询放大为多组 OD 查询，因此不能使用“全国站点全量 DFS 枚举”。真实 provider 的中转能力必须满足：

- 单次搜索设置最大外部请求预算，例如 `max_remote_queries`。
- 所有 OD 查询先查本地缓存，缓存未命中且仍有预算时才请求 12306。
- 超出请求预算后，只允许使用本地缓存或跳过该候选，不继续扩大外部请求。
- 搜索结果必须可降级：没有中转结果时仍返回直达结果；真实数据源失败时不影响 Mock provider。

### 候选中转站生成

候选站生成不应依赖票价接口实时暴搜，而应基于本地站点知识库和启发式评分：

1. **空间剪枝**：车站需要落在出发站和目的站连线附近的动态带状区域内。
   - 计算站点到 AB 线段的垂直距离。
   - 阈值可取 `max(50km, min(300km, 0.2 * AB距离))`，后续可按实测调整。
2. **朝向剪枝**：计算候选站在 AB 方向上的投影。
   - `projection < 0` 或 `projection > AB长度` 的站点直接剔除。
3. **枢纽优先**：对车站维护 `centrality_score`，例如经停车次数、出边数量、人工维护的核心枢纽等级。
4. **铁路拓扑优先**：后续建立本地铁路图后，候选站排序从单纯 GPS 距离升级为 `GPS距离 + 铁路拓扑距离 + 枢纽分` 的综合评分。

候选中转站建议先限制在 Top 20 到 Top 50，再由请求预算决定实际查询数量。

当前站点知识库已从外部站点数据集导入项目内 CSV：

```text
backend/app/data/station_metadata.csv
```

导入脚本：

```text
backend/scripts/import_station_metadata.py
```

默认来源为本机外部数据目录：

```text
D:\AbelTomato_Files\Developer\Others\China-rail-way-stations-data\src\station.csv
```

导入时只保留 `性质 == 客运站` 且存在合法 WGS84 坐标的站点，输出字段包括站名、经纬度、中心性分数、省市、铁路局和站点类型。外部数据源不包含 12306 电报码，因此 `telecode` 暂为空；真实 12306 查询仍应由 provider 维护站名到电报码的映射。

当前导入后的站点知识库约 1900 个客运站。`StationMetadataRepository` 默认优先读取该 CSV；如果 CSV 缺失或不可读，则回退到代码内置的小型站点列表，保证测试和基础功能可降级运行。

注意：该外部数据源 README 标注数据可能过期，因此它只作为候选站空间知识库，不作为票价或实时车站有效性的唯一事实源。`LOW_PRICE_TRANSFER_HUBS` 和 `CHEAP_TRANSFER_SCORES` 仍保留，用于表达低价断点和中转枢纽的人工先验。

### 最低价导向的启发式剪枝

当前项目不做最快或综合排序，因此不采用可变权重 `Cost = α*Price + β*Time + ...` 作为主排序。核心目标应保持为：

```text
min(total_price)
```

在最低价目标下，可使用以下剪枝：

1. **直达价格上界**：如果直达最低价为 `best_price`，任何已知第一段价格 `first_price >= best_price` 的候选不再扩展。
2. **中转组合剪枝**：组合两段时，若 `first_price + second_price >= best_price`，且只需要严格更低价方案，可跳过；若需要返回 Top K，则保留但降低优先级。
3. **时间可行性剪枝**：第二段出发时间必须晚于第一段到达时间加最小换乘时间。
4. **总耗时剪枝**：超过 `max_total_duration_minutes` 的组合直接剔除。
5. **请求预算剪枝**：优先查询更可能形成低价方案的候选站，预算耗尽后停止远程请求。

### 缓存与本地知识库

缓存粒度应与 12306 接口一致，缓存对象是 OD 查询结果，而不是单个车次：

```text
provider + date + from_station + to_station -> list[TrainSegment]
```

当前已实现内存 TTL 缓存和可选 SQLite 持久化缓存：

```text
query_cache(
  provider,
  travel_date,
  origin,
  destination,
  fetched_at,
  expires_at,
  response_json
)
```

后续可逐步建立本地铁路知识库：

- `station`：站名、电报码、经纬度、枢纽分。
- `train_stop`：车次、站序、到发时间。
- `edge`：区间、车次、历时、历史票价。

最终目标是：路线搜索尽量在本地知识库完成，只对少量候选 OD 做在线票价刷新。
