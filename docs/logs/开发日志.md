## 2026-06-26 中转搜索算法与真实数据源稳定性优化

### 已完成成果

#### 设计文档

- 新增 `docs/中转搜索算法优化路线.md`，记录从当前中转搜索到 Best First Search / A* 的完整演进路线。
- 明确项目长期定位为“全国铁路最低票价路径搜索器”，搜索主目标保持为最低总票价。
- 明确历史最低票价通常是上界，不应直接作为 A* admissible lower bound 用于硬剪枝。

#### 一换搜索优化

- `RouteSearchService` 一换搜索增加第一段为空剪枝：`A -> T` 无车次时不再查询 `T -> B`。
- 换乘组合从双层暴力枚举调整为按第二段发车时间排序并用二分定位可换乘车次。
- 中转方案支持可选 `max_total_duration_minutes` 过滤；当前默认值改为 `None`，即不再默认限制方案总耗时，只有调用方显式传入分钟数时才过滤。
- 搜索结果仍按 `total_price` 优先排序，总耗时和换乘次数只作为同价稳定排序条件。

#### 受控并发与请求预算

- `RouteSearchService` 新增 `max_concurrent_remote_queries`，使用 `asyncio.Semaphore` 控制真实远程查询并发。
- 使用 `asyncio.Lock` 保护 `remote_query_count` 检查和递增，避免并发下超出 `max_remote_queries`。
- `/api/providers/status` 增加 `max_concurrent_remote_queries` 诊断字段。
- 前端状态面板同步展示 OD 请求预算、并发上限和上次实际请求数。

#### 候选站低价评分

- `CandidateTransferStationGenerator` 新增 `CHEAP_TRANSFER_SCORES` 静态低价断点评分。
- 候选站排序从几何距离优先调整为：低价断点评分、铁路中心性、走廊垂距、站名。
- 保留地理走廊过滤，避免候选站无约束扩散。

#### 12306 错误诊断与候选容错

- 新增通用 `TrainDataProviderError`，`Railway12306Error` 继承该错误类型。
- 新增 `describe_httpx_error()`，让 12306 请求失败时包含异常类型、HTTP 状态、请求 URL 或超时说明，避免只显示空的“票价接口请求失败：”。
- 一换候选站单个 OD 请求失败时跳过该候选，不再中断整次搜索。
- 使用探测脚本确认 `leftTicketPrice/queryAllPublicPrice` 当前可访问，但部分 OD 可能返回 0 车次或偶发失败。

#### `best_price` 剪枝

- 新增 `BestPriceTracker`，在并发候选任务间共享当前已知最低价。
- 直达方案会先更新 `best_price`，后续中转搜索可基于该价格剪枝。
- 当第一段票价已不可能优于当前最低价时，不再查询或组合后续方案。
- 当完整中转方案总价不低于当前最低价时跳过该方案。

#### A* / Uniform Cost Search 搜索器

- 新增 `PathSearchState`，表示 A* 状态：当前站、已选车次、累计票价、到达时间和已访问站点。
- 新增 `CheapestPathSearcher`，使用优先队列按 `g + h` 扩展路径。
- 当前 `_heuristic_lower_bound()` 返回 `0`，因此实现等价于 Uniform Cost Search，优先保证最低价正确性。
- 当 `RouteQuery.max_transfers >= 2` 时启用新搜索器，支持两次中转搜索。
- 保留 `max_transfers = 0/1` 的原优化路径，降低回归风险。

#### Pareto frontier 支配剪枝

- 新增 `FrontierEntry` 和 `ParetoFrontier`。
- 状态只有在已有状态“累计价格不高且到达时间不晚”时才会被剪枝。
- 保留“更贵但更早到达”的状态，避免因忽略时间维度误剪可行低价后续路径。

#### OD SQLite 持久化缓存

- `RouteSearchService` 新增可选 `persistent_cache`。
- 支持通过环境变量启用默认 SQLite OD 缓存：
  - `ROUTE_SEGMENT_CACHE_DB`
  - `ROUTE_SEGMENT_CACHE_TTL_SECONDS`
- `_search_segments()` 查询顺序调整为：内存 TTL 缓存 -> SQLite OD 缓存 -> 远程 Provider。
- SQLite 命中会回填内存缓存，并且不增加 `remote_query_count`。
- 新增 `serialize_train_segments()` 和 `deserialize_train_segments()`，使用 Pydantic JSON 模式序列化 `TrainSegment`。

#### 搜索遥测与历史命中记录

- 新增 `SqliteSearchTelemetryRecorder`，持久化记录每次搜索：
  - 查询时间。
  - provider。
  - 出发地、目的地、出行日期。
  - 最大中转次数、最短换乘时间、最大总时长。
  - 方案数量、最佳价格、最佳中转站、远程请求数。
- 新增 `transfer_station_hits` 表，累计某 OD 下中转站命中次数和历史最佳价格。
- `RouteSearchService` 新增可选 `telemetry_recorder`。
- 支持通过环境变量启用默认 SQLite 遥测库：`SEARCH_TELEMETRY_DB`。

### 待解决问题：东莞到拉萨方案搜不到

现象：

- 查询 `东莞 -> 拉萨` 仍返回空方案；该问题与方案总耗时限制无关，取消默认 `max_total_duration_minutes` 后仍存在。

已定位原因：

- 当前未设置 `TRAIN_DATA_PROVIDER` 时默认使用 `mock` provider；Mock 数据只覆盖北京、济南西、南京南、上海、天津南、杭州东，不可能返回东莞到拉萨。
- 本地 `data/static_prices.sqlite3` 静态库没有任何 `destination = '拉萨'` 的 OD 记录；`郑州/西安/兰州/西宁 -> 拉萨` 也均未覆盖。
- 12306 公布票价接口可查到部分进藏段，例如 `郑州 -> 拉萨`、`西安 -> 拉萨`、`兰州 -> 拉萨`、`西宁 -> 拉萨`。
- 12306 公布票价接口也可查到 `东莞 -> 郑州`、`东莞 -> 西安`，因此理论上存在一换路径 `东莞 -> 郑州 -> 拉萨` 或 `东莞 -> 西安 -> 拉萨`。
- 当前候选站生成器对 `东莞 -> 拉萨` 的 Top 50 候选主要集中在华南、西南走廊，如广州、衡阳、株洲、韶关东、柳州、南宁、昆明、贵阳等，未包含 `郑州`、`西安`、`兰州`、`西宁` 等进藏关键中转站，导致真实 provider 下也不会尝试这些可行中转点。

后续处理方向：

1. 对目的地为 `拉萨` 的查询增加战略中转站强制注入：`郑州`、`西安`、`兰州`、`西宁`。
2. 将上述站点纳入低价/战略中转 hub，并提高评分，确保能进入候选 Top N。
3. 刷新静态库缺失 OD：`东莞 -> 郑州`、`郑州 -> 拉萨`、`东莞 -> 西安`、`西安 -> 拉萨`，可选补充 `兰州 -> 拉萨`、`西宁 -> 拉萨`。
4. 验证真实 provider 查询时，`东莞 -> 拉萨` 至少能返回经郑州或西安的一换方案。

### 下一步建议

1. 用 `transfer_station_hits` 反哺候选站排序：历史命中次数高、历史最佳价格低的中转站优先扩展。
2. 保持 A* `h=0`，先只改变候选扩展顺序，不改变硬剪枝规则。
3. 后续若要实现真正启发函数，必须先设计不会高估的 lower bound 来源。

## 2026-06-26 最低价结果体验排序与后续瓶颈判断

### 用户体会

- 当前目标不是泛化推荐路线，而是严格围绕“最低总价”筛选结果。
- 同价方案不应简单按总耗时排序；中途等待时间会明显影响体验，应作为同价排序依据。
- 返回结果不需要过多，保留 `10` 个方案足够用于前端展示和用户比较。
- 真正的性能瓶颈大概率不在本地搜索算法，而在真实数据源远程 I/O。
- 后续更重要的方向是实现数据库持久化，记录站点或 OD 历史最低价及更新时间。铁路公布票价相对稳定，可以较长时间不更新，用于减少重复外部请求。

## 2026-06-30 候选中转站算法重新定义

### 背景判断

当前 `CandidateTransferStationGenerator` 使用地理走廊剪枝、中心性分数和 `LOW_PRICE_TRANSFER_HUBS` / `CHEAP_TRANSFER_SCORES` 人工先验生成候选中转站。该实现能控制真实数据源请求量，但不具备全局最优保证。

本次重新评估后，确认“低价站点”不是严格成立的铁路票价概念。铁路票价主要由计价里程、列车等级、车型、席别、线路计价规则和折扣等因素决定。某个车站本身不会天然产生低价；它最多可能作为换乘枢纽、径路分叉点、车型切换点或铁路通道节点，使 `A -> T + T -> B` 的组合价格低于 `A -> B`。

因此，后续文档和代码不应继续把候选站称为“低价站点”或“低价断点”。更准确的概念是：

- 换乘枢纽先验。
- 铁路通道先验。
- 车型/线路切换先验。
- 历史命中先验。

### 当前算法正确性边界

现有几何走廊剪枝可能直接剪掉可行方案甚至最优方案。典型场景是 `东莞 -> 拉萨`：

- 按地理直线走廊，候选站容易集中在华南、西南方向。
- 真实可行中转可能需要绕行到 `郑州`、`西安`、`兰州`、`西宁` 等铁路干线节点。
- 如果这些节点没有进入候选集合，后续一换或二换搜索不会查询相关 OD，即使真实 provider 可以查到票价，也会返回无方案。

结论：当前候选生成是启发式召回，不是正确性剪枝。它只能在受控请求预算下提高可用性，不能声称覆盖全国全局最低价。

### 新目标

候选中转站生成应从“内部写死的几何 Top N”调整为“用户可控的搜索范围参数 + 多策略召回”。系统目标改为：

```text
在用户指定的候选数量、绕行范围、最大中转次数和请求预算内，尽量搜索最低总价方案，并清晰暴露搜索范围限制。
```

界面和 API 不应承诺“保证全国最低价”，除非未来建立完整铁路图、稳定 OD 覆盖和可证明不会高估的 lower bound。

### API 参数草案

后续可在 `RouteQuery` 增加搜索控制字段：

```python
candidate_limit: int = 50
candidate_strategy: Literal["balanced", "direct_corridor", "wide_detour", "hub_first", "exhaustive_budgeted"] = "balanced"
max_detour_ratio: float | None = None
max_detour_km: int | None = None
corridor_width_km: int | None = None
```

既有字段继续保留：

```python
max_transfers: int
min_transfer_minutes: int
max_total_duration_minutes: int | None
```

前端高级搜索可映射为：

- 最大中转次数：`0 / 1 / 2`。
- 搜索深度：快速、标准、深度、极深。
- 允许绕行：少、中、多、不限。
- 候选站上限：例如 `20 / 50 / 100 / 200`。
- 最大总耗时：可选输入。

### 候选生成新方案

候选集合应由多种策略合并去重，而不是只依赖 AB 直线走廊：

1. 地理走廊候选：保留现有投影和垂距逻辑，但走廊宽度由用户参数或策略派生。
2. 绕行约束候选：允许满足 `distance(A,T) + distance(T,B) <= distance(A,B) * (1 + max_detour_ratio)` 的站点进入候选。
3. 高枢纽候选：按 `centrality_score` 选择全国或区域高连接能力站点。
4. 战略通道候选：对拉萨、新疆、东北边境、西南边境等稀疏目的地注入铁路通道节点，例如进藏方向的 `郑州`、`西安`、`兰州`、`西宁`、`格尔木`。
5. 历史命中候选：把 `transfer_station_hits` 中曾产生可行或更优方案的中转站前置排序。
6. 用户显式候选：后续可允许用户指定必须尝试的中转站。

排序语义应改为：

```text
历史命中优先级
铁路枢纽/通道先验
绕行代价
走廊垂距
站名稳定排序
```

### 文档和代码后续处理

1. 将 `LOW_PRICE_TRANSFER_HUBS` 重命名为 `STRATEGIC_TRANSFER_HUBS` 或 `MANUAL_TRANSFER_HUBS`。
2. 将 `CHEAP_TRANSFER_SCORES` 重命名为 `TRANSFER_PRIOR_SCORES` 或 `CONNECTIVITY_PRIOR_SCORES`。
3. 将 `cheap_transfer_score` 改为 `transfer_prior_score`，避免继续表达“站点天然低价”。
4. 一换搜索也应使用历史命中排序，目前历史命中主要影响 `max_transfers >= 2` 的 `CheapestPathSearcher`。
5. API 响应或流式 metadata 应暴露候选数量、已扩展候选、失败候选、请求预算是否耗尽和搜索参数。
6. 增加 `东莞 -> 拉萨` 这类回归测试，验证宽松绕行策略能召回进藏通道节点。

## 2026-06-30 时间扩展图搜索方案记录

### 背景

针对 `厦门 -> 深圳`、`2026-07-12`、`max_transfers=2`、`static-with-remote-fallback` 的排查确认：

- `赣州` 已进入候选站，且排序靠前。
- `厦门 -> 赣州` 本地静态库有数据，最低价 `62.5`。
- `赣州 -> 深圳` 本地静态库缺少 `2026-07-12/2026-07-13` 数据。
- fallback 模式理论上应补查该 OD，但当前二换 A* 搜索会先消耗大量 OD 查询预算，导致关键 OD 没有机会触发 fallback。

因此，问题不只是候选站召回，而是当前“候选站 + OD 试探 + 请求预算”的搜索模型无法稳定保证关键边被优先补查。

### 新方案

新增 `docs/时间扩展图搜索方案.md`，记录后续搜索架构方向：

```text
离线/准实时构建本地时间扩展图，在线执行 Dijkstra 或 label-setting 最短路搜索。
```

核心思想：

1. 以车站时间事件作为节点。
2. 以车次 OD 或经停区间作为乘车边，边权为票价。
3. 同站到达事件连接到满足最短换乘时间的第一个出发事件。
4. 同站事件按时间线性连接等待边，避免 `O(n^2)` 换乘边。
5. 搜索阶段用价格作为主权重，并携带换乘次数、总耗时等约束状态。

### 分阶段路线

1. 阶段一：基于现有 `od_public_price_snapshot` 构建本地时序 OD 图，先验证 Dijkstra 搜索结构。
2. 阶段二：在建图前做受控缺边补全，把 fallback 用于高价值缺失 OD，而不是在搜索过程中无序消耗预算。
3. 阶段三：如果新数据源支持完整车次经停和同车 OD 票价，再升级为完整经停时间扩展图。

### 当前约束

- 现有静态库是 OD 票价快照，不是完整经停运行图。
- 在没有完整经停和相邻区间票价之前，不能随意把 OD 票价拆成相邻站边权。
- 候选站模块短期仍可保留，但应从“限制搜索空间”转为“缺边补全优先级”和“fallback 预算控制”。

### 下一步

等待测试新数据源，重点确认其是否能提供：

1. 完整车次经停序列。
2. 到发时刻。
3. 同车任意 OD 票价或相邻区间票价。
4. 查询日、次日、第三日覆盖。
5. 可接受的本地索引和更新方式。

## 2026-06-30 多重搜索架构决策记录

### 背景

已阅读 `docs/时间扩展图搜索方案.md` 并纵览当前项目结构。现有在线路线搜索入口集中在：

- `backend/app/api/routes.py`
- `backend/app/services/route_search.py`
- `backend/app/adapters/base.py`
- `backend/app/adapters/provider_factory.py`
- `backend/app/services/static_price_repository.py`

当前 `RouteSearchService` 同时承担：

1. API 搜索服务职责。
2. 候选中转站搜索算法职责。
3. OD 查询缓存、请求预算、并发限速、搜索遥测职责。
4. 对 `TrainDataProvider.search_segments()` 的调度职责。

如果直接把时间扩展图搜索写进 `RouteSearchService`，会导致旧 OD 试探搜索、静态库 fallback、缓存遥测和新图搜索互相耦合，后续难以回退、测试和验证。

### 决策

不直接替换现有搜索实现。采用多重搜索架构：

```text
API 层
  ↓
RouteSearchEngine 抽象
  ├─ candidate：现有候选中转站 + OD 试探搜索
  └─ time-expanded：本地时间扩展图搜索
```

环境变量控制搜索引擎选择：

```text
ROUTE_SEARCH_ENGINE=candidate       # 默认，保持旧行为
ROUTE_SEARCH_ENGINE=time-expanded   # 新本地时间扩展图搜索
```

`TRAIN_DATA_PROVIDER` 继续控制旧搜索所用数据源：`mock`、`12306-public-price`、`static-price`。新时间扩展图搜索不应依赖在线 provider 远程查询，而应只读本地 SQLite 中的 `train_stop_snapshot` 和 `train_od_price_snapshot`。

### 设计边界

#### 保留旧搜索

- 旧 `RouteSearchService` 暂不删除。
- 旧搜索继续支持 `static-with-remote-fallback`，但该模式只属于旧 OD 搜索链路。
- `CandidateTransferStationGenerator` 继续保留，可用于旧搜索、离线补采优先级、数据覆盖提示。

#### 新增搜索抽象

新增统一搜索引擎接口，建议文件：

```text
backend/app/services/route_search_engine.py
backend/app/services/route_search_engine_factory.py
```

接口语义保持贴近 API 当前需要：

```python
class RouteSearchEngine(Protocol):
    @property
    def source(self) -> str: ...

    @property
    def status(self) -> dict[str, object]: ...

    async def search(self, query: RouteQuery) -> RouteSearchResponse: ...

    async def stream_snapshots(self, query: RouteQuery) -> AsyncIterator[StreamSnapshot]: ...
```

#### 新时间扩展图搜索约束

- 在线查询阶段禁止调用远程接口。
- 不调用 `TrainDataProvider.search_segments()`。
- 本地数据缺失时返回空方案或覆盖不足诊断，不进行 fallback 补查。
- 不把长 OD 票价拆成相邻区间累加。
- 只有 `train_od_price_snapshot` 存在同车合法 OD 票价时，才允许建立乘车边。

### 执行计划文档

已决定新增原子化执行文档：

```text

```

该文档应面向能力较低的执行模型，每个切片必须足够小、完整、可验证，并包含：

1. 修改文件。

### 验收口径

第一阶段不要求完整离线采集链路。验收重点是架构边界成立：

1. 默认行为仍走旧搜索。
2. 环境变量可切换到新时间扩展图搜索。
3. 新搜索路径在线不访问远程 provider。
4. 新搜索路径有本地 SQLite 样例测试。
5. API 层不感知具体搜索算法实现。

## 2026-07-02 相邻区间票价采集切片 3：station OD 去重请求

### 已完成成果

- `crawl_time_expanded_full_snapshot.py` 新增 `group_price_tasks_by_station_od()`，按 `(from_station, to_station)` 对 `PriceTask` 分组。
- 实际票价采集阶段从“逐 train OD 请求”调整为“逐 unique station OD 请求”：

- station OD 级请求失败时，按该 OD 分组下的 task 数批量计入 `price_skipped` 或 `price_failed`，避免请求级失败被低估。

## 2026-07-02 相邻区间票价采集切片 4：相邻累加误差验证脚本

### 已完成成果

- 新增 `backend/scripts/compare_adjacent_vs_direct_price.py`，用于抽样比较 direct OD 公布票价与相邻段累加票价误差。
- 脚本从本地 `train_stop_snapshot` 按 `provider + service_date` 抽样车次，依赖切片 1 已落库的真实时刻表快照。

- 使用 `train_no` / `train_code` 与席别对齐 direct rows 和 adjacent rows。
- 计算 `adjacent_sum_price - direct_price`、绝对误差，并按席别汇总误差统计。
- 输出 JSON 报告字段：
  - `sample_count`
  - `exact_match_count`
  - `diff_count`
  - `avg_abs_diff`
  - `max_abs_diff`
  - `by_seat_type_diff_stats`
- 输出前 20 条 comparison samples，便于人工检查具体车次、OD、席别和误差。
