## 4. 原子修复切片

### S01：移除被跟踪的运行时 SQLite 数据库

目标：消除仓库污染，保持 `data/*.sqlite3` 只作为本地运行数据。

改动范围：

- Git 索引：移除 `data/static_prices.sqlite3`。
- 文档：`README.md` 或 `docs/项目整体架构.md` 补充如何生成或刷新本地静态库。

建议操作：

```powershell
git rm --cached data/static_prices.sqlite3
```

测试与验证：

- `git status --short` 显示 `D  data/static_prices.sqlite3`，本地文件仍存在。
- `git ls-files data/static_prices.sqlite3` 无输出。
- 后端测试无需因该切片全量运行；若文档同步改动，可只审查 diff。

验收标准：

- 仓库不再跟踪运行时 SQLite。
- `.gitignore` 保持忽略 `data/*.sqlite3`。
- README 或架构文档说明静态库生成方式。

风险：

- 如果某些测试隐式依赖该数据库，需要改为使用测试 fixture 或临时库。当前测试基线未发现此依赖。

### S03：抽离每次搜索状态，解决并发串扰

目标：让并发搜索互不重置诊断和请求计数。

改动范围：

- `backend/app/services/route_search.py`
- `backend/app/api/routes.py`
- `backend/tests/test_route_search.py`
- `backend/tests/test_api.py`

设计方向：

- 新增 `SearchContext`，包含：
  - `remote_query_count`
  - `diagnostics`
  - `best_price` 或一换搜索需要的状态
- `RouteSearchService` 保留：
  - provider
  - memory cache
  - persistent cache
  - semaphore / 配置
  - telemetry recorder
- `search()` / `stream()` 每次创建自己的 `SearchContext`。
- `_search_segments()` 接收 context，而不是读写 `self.remote_query_count` / `self.last_diagnostics`。
- `last_diagnostics` 如果仍要暴露给 `/providers/status`，只在搜索完成时用 lock 写入一次快照。

测试：

- 新增并发测试：同时发起两个搜索，断言两个搜索各自的 `remote_query_count` 不互相覆盖。
- 保留现有 provider status 测试，必要时改为断言“最近一次完成搜索”的快照。

验证命令：

```powershell
cd backend
.\.venv\Scripts\python.exe -m pytest -q
```

验收标准：

- 后端测试全通过。
- 并发搜索测试能稳定复现并覆盖旧问题。
- `RouteSearchService` 不再在搜索中直接重置共享的 `remote_query_count` / `last_diagnostics`。

风险：

- 该切片触及核心搜索链路，必须保持小步提交。
- 不要同时改搜索算法排序逻辑，避免混入行为变化。

### S04：provider 初始化改为可诊断状态

目标：配置错误时服务可启动，健康检查和状态接口能说明问题。

改动范围：

- `backend/app/api/routes.py`
- `backend/app/main.py`
- `backend/app/adapters/provider_factory.py`
- `backend/tests/test_api.py`

设计方向：

- 引入 `ProviderRuntime` 或 `AppState`：
  - `provider: TrainDataProvider | None`
  - `route_search_service: RouteSearchService | None`
  - `startup_error: str | None`
- FastAPI lifespan/startup 中初始化 provider。
- `/api/health` 在初始化失败时返回 `status: degraded` 或 HTTP 503，二者择一并固定。
- `/api/providers/status` 返回 `configured/degraded/unavailable`。
- `/api/routes/search` 在 provider 不可用时返回 503。

测试：

- monkeypatch 非法 `TRAIN_DATA_PROVIDER`，验证 app 不在 import 时崩溃。
- 验证搜索端点返回 503。

验收标准：

- 配置错误不会导致 `from app.main import app` 直接失败。
- 错误响应结构可被前端识别。

风险：

- FastAPI TestClient 与 lifespan 行为要明确，测试需覆盖启动流程。

### S06：定义流式 NDJSON 事件协议

目标：让流式查询的成功、错误、元信息语义明确。

改动范围：

- `backend/app/api/routes.py`
- `frontend/src/lib/api.ts`
- `docs/项目整体架构.md`
- 后端 API 测试、前端 API 测试（如果 S14 已落地）

设计方向：

每行 NDJSON 改为事件对象：

```json
{"type":"meta","query_id":"...","source":"...","updated_at":"..."}
{"type":"plan","plan":{}}
{"type":"error","message":"...","category":"data_source_unavailable"}
{"type":"done"}
```

注意：

- 如果错误发生在响应头发送后，HTTP 状态仍可能是 200，所以必须文档化事件级错误。
- 如果错误可在流开始前预检发现，可直接返回 HTTP 4xx/5xx。

测试：

- 前端解析 `meta/plan/error/done`。
- 后端 fake provider 产生错误事件。

验收标准：

- 流式 `updated_at` 不再由前端伪造。
- 前端对未知事件和非法 JSON 有明确错误提示。

### S07：增强 `RouteQuery` 输入校验

目标：减少无意义请求、缓存污染和真实数据源浪费。

改动范围：

- `backend/app/domain/models.py`
- `backend/tests/test_*`
- 如前端需同步提示，则改 `frontend/src/App.tsx`

建议规则：

- `from_station` / `to_station`：strip 空白，`min_length=1`，`max_length=32` 或 `64`。
- 起点和终点不能相同。
- `date`：按产品选择限制范围。
  - 若真实 12306：不早于今天，不超过可查询窗口。
  - 若保留 Mock 测试历史日期：把日期范围限制放到 provider 或 API 层，避免破坏固定日期测试。
- `max_total_duration_minutes` 增加合理上限，例如 `le=72*60`。

测试：

- 空白被 trim。
- 起终点相同返回 422。
- 超长站名返回 422。
- 非法日期按约定返回 422 或 provider 受控错误。

验收标准：

- 后端测试全通过。
- 前端对 422 仍显示“查询参数错误”。

## 6. 每个切片的完成定义

每个修复切片完成时必须满足：

1. 只解决一个主要问题，不混入无关重构。
2. 更新或新增对应测试；纯文档/仓库卫生切片除外。
3. 若行为、接口、环境变量、启动方式变化，必须同步 README 或架构文档。
4. 至少运行：
   - 后端相关切片：`cd backend; .\.venv\Scripts\python.exe -m pytest -q`
   - 前端相关切片：`pnpm --dir frontend build`
5. 在提交说明中写明对应切片 ID，例如 `S03: isolate route search request state`。

## 7. 不建议合并处理的事项

以下事项不要放进同一个提交：

- S03 并发状态重构 与 S06 流式协议变更。
- S07 输入校验 与前端 UI 大改。
- S11 SQLite I/O 治理 与 repository schema 变更。
- S12 依赖固定 与 Vitest 引入。

原因：这些组合会同时改变行为、依赖和测试基线，回归定位困难。
