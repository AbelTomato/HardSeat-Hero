# 数据源调研

## 结论

第一阶段不应把官方实时数据作为核心依赖。建议采用数据源适配器架构，先用 Mock 数据完成算法和界面，再调研并接入一个真实数据源。

## 候选方向

### GitHub 开源项目

调研重点：

- 是否仍在维护。
- 是否提供可直接调用的接口或可复用的数据模块。
- 数据来源是否清晰。
- 是否需要登录态、Cookie、验证码或特殊请求头。
- 许可证是否允许复用。
- 是否有近期 issue 显示接口失效。

风险：

- 项目可能封装的是非公开接口，稳定性不可控。
- 接口变更后维护成本较高。
- 许可证可能限制商业或二次分发。

### MCP 查询项目

调研重点：

- MCP 工具是否能输入出发地、目的地、日期并返回结构化车次数据。
- 返回格式是否稳定。
- 是否依赖本地浏览器、登录态或外部服务。
- 查询失败时是否有明确错误。

建议：

- MCP 只作为数据源适配器，不应直接进入核心算法。
- 需要把 MCP 返回结果转换为统一的 `TrainSegment` 模型。

### 静态站点数据与价格样例

用途：

- 用于开发、单元测试和算法验证。
- 用于真实数据源不可用时的演示降级。

限制：

- 不能代表真实票价。
- 不能表达余票变化。

## 数据源适配策略

统一抽象：

```text
RouteSearchService
  -> TrainDataProvider
       -> MockTrainDataProvider
       -> GithubTrainDataProvider
       -> McpTrainDataProvider
```

统一输出字段：

- 车次号。
- 出发站。
- 到达站。
- 出发时间。
- 到达时间。
- 历时分钟。
- 席别价格。
- 余票状态。
- 数据来源。
- 更新时间。

## 缓存建议

- 缓存 key：`provider:date:from_station:to_station`。
- MVP TTL：5 到 15 分钟。
- 缓存内容：归一化后的 `TrainSegment` 列表。
- 后续可用 Redis 替代内存缓存。
