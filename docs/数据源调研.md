# 数据源调研

## 结论

当前已采用数据源适配器架构，并已接入 `Railway12306PublicPriceProvider`、`LocalStaticPriceProvider` 和 `StaticPriceFallbackProvider`。本文保留早期调研结论和风险说明；当前实现状态以 `docs/项目整体架构.md` 为准。

## 候选方向

### 12306 公布票价接口

结论：`leftTicketPrice/queryAllPublicPrice` 可作为真实票价的首选探测方向，但不能作为强稳定依赖。

可行链路：

- `station_name.js`：获取站名和电报码映射。
- `queryPublicIndex.html`：12306 官方公布票价查询页面。
- `leftTicketPrice/queryAllPublicPrice`：按日期、出发站电报码、到达站电报码查询车次、时刻和公布票价。
- `queryByTrainNo`：补充经停车站和时刻表详情。

已验证：

- 北京南到上海虹桥可返回 G/D 类车次和商务座、一等座、二等座价格。
- 北京到广州、上海到成都等 OD 可返回 K/Z 普速车的硬座、硬卧、软卧和高级软卧价格。
- 该接口直接返回 JSON，不需要模拟浏览器填表。

用户提供的旧示例代码验证了 `leftTicket/queryZ` 与 `queryTicketPrice` 链路的大致思路，但存在以下问题，不能直接引入项目：

- `if d == '9MO' or 'OM9'` 条件永远为真，席别分支会错误命中。
- 余票字段不等于价格，必须以后续 `queryTicketPrice` 返回为准。
- 按 `seat_types` 写死分支覆盖不完整，应按票价接口返回 key 通用映射。
- 全局 `PrettyTable`、交互式输入、本地 CSV、缺少超时和错误分类，不适合后端服务。
- 当前环境下直接请求 `leftTicket/queryZ`、`leftTicket/query`、`leftTicket/queryA` 均返回 HTML，不是 JSON。

建议正式接入时拆成 `StationCodeRepository` 和 `PublicPriceClient`，并通过 `TrainDataProvider` 输出统一 `TrainSegment`。

### GitHub 开源项目（未采用）

调研重点：

- 是否仍在维护。
- 是否提供可直接调用的接口或可复用的数据模块。
- 数据来源是否清晰。
- 是否需要登录态、Cookie、验证码或特殊请求头。
- 许可证是否允许复用。
- 是否有近期 issue 显示接口失效。

风险：

- 项目可能封装的是非公开接口，稳定性不可控。
- 接口变更后维护成本较高。
- 许可证可能限制商业或二次分发。

### MCP 查询项目（未采用）

调研重点：

- MCP 工具是否能输入出发地、目的地、日期并返回结构化车次数据。
- 返回格式是否稳定。
- 是否依赖本地浏览器、登录态或外部服务。
- 查询失败时是否有明确错误。

建议：

- MCP 只作为数据源适配器，不应直接进入核心算法。
- 需要把 MCP 返回结果转换为统一的 `TrainSegment` 模型。

### 静态站点数据与价格样例

用途：

- 用于开发、单元测试和算法验证。
- 用于真实数据源不可用时的演示降级。

限制：

- 不能代表真实票价。
- 不能表达余票变化。

## 数据源适配策略

统一抽象：

```text
RouteSearchService
  -> TrainDataProvider
       -> MockTrainDataProvider
       -> Railway12306PublicPriceProvider
       -> LocalStaticPriceProvider
       -> StaticPriceFallbackProvider
```

统一输出字段：

- 车次号。
- 出发站。
- 到达站。
- 出发时间。
- 到达时间。
- 历时分钟。
- 席别价格。
- 余票状态。
- 数据来源。
- 更新时间。

## 缓存建议

- 缓存 key：`provider:date:from_station:to_station`。
- MVP TTL：5 到 15 分钟。
- 缓存内容：归一化后的 `TrainSegment` 列表。
- 后续可用 Redis 替代内存缓存。
- 真实中转搜索应优先落地 SQLite OD 持久化缓存，减少重复查询 12306。
- 单次搜索需要请求预算，预算耗尽后只读取本地缓存，不继续触发外部请求。

## 中转候选站数据建议

真实中转能力需要本地站点元数据，不应依赖实时接口全量枚举。

候选站元数据至少包含：

- 站名。
- 电报码。
- 经纬度。
- 枢纽分，例如经停车次数、出边数量或人工维护等级。

候选站生成策略：

- 使用出发站和目的站连线的带状区域做空间剪枝。
- 使用投影判断候选站是否位于行进方向上。
- 使用枢纽分和距离综合排序，优先查询更可能形成低价中转的站点。
- 后续建立铁路拓扑图后，用铁路拓扑距离替代或补充 GPS 距离。
