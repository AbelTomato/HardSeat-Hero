# 实施计划

## 阶段 1：项目骨架

- 初始化 `backend/` FastAPI 项目。
- 初始化 `frontend/` React + Vite + TypeScript 项目。
- 配置 Tailwind CSS 和 shadcn/ui。
- 添加根目录 README，说明启动方式和项目目标。

验收标准：

- 后端 `GET /api/health` 可返回正常状态。
- 前端开发服务器可启动并显示基础页面。

## 阶段 2：领域模型与 Mock 数据

- 定义 Pydantic schema 和领域模型。
- 定义 `TrainDataProvider` 抽象接口。
- 实现 `MockTrainDataProvider`。
- 准备覆盖直达、中转、无票价、跨日场景的样例数据。

单元测试：

- schema 校验测试。
- Mock 数据源查询测试。

验收标准：

- 后端可以通过 Mock 数据返回指定站点之间的车次段。

## 阶段 3：中转搜索算法

- 实现直达方案构建。
- 实现 1 次中转方案枚举。
- 实现最短换乘时间过滤。
- 实现总价和总耗时计算。
- 实现按最低价排序。

单元测试：

- 直达方案返回正确。
- 一次中转方案返回正确。
- 换乘时间不足的方案被过滤。
- 无可用方案时返回空列表。
- 总价排序稳定。

验收标准：

- `POST /api/routes/search` 可返回按价格排序的方案列表。

## 阶段 4：前端查询体验

- 实现出发地、目的地、日期输入。
- 实现查询提交和加载状态。
- 实现方案列表展示。
- 实现错误和空状态。

单元测试或组件测试：

- 搜索表单提交参数正确。
- 结果列表渲染车次段、总价、总耗时。
- 空结果和错误状态可见。

验收标准：

- 用户可以在浏览器中完成一次查询并查看结果。

## 阶段 5：真实数据源调研与接入

- 调研 GitHub 上可用的火车票查询项目。
- 记录数据来源、许可证、更新频率、调用限制、失效风险。
- 选择一个数据源实现适配器。
- 增加数据源状态接口。
- 增加缓存和失败降级。

集成测试：

- 数据源正常时返回真实格式归一化后的车次段。
- 数据源失败时返回受控错误，不影响服务进程。

验收标准：

- 系统可以在 Mock 数据源和真实数据源之间切换。

## 阶段 6：交付前验证

- 执行后端单元测试。
- 执行前端构建。
- 执行基础联调。
- 更新 README 和数据源调研文档。

验收标准：

- 测试通过。
- 前后端启动说明完整。
- 已记录真实数据源限制和风险。
