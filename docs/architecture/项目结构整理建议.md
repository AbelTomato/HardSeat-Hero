# 项目结构整理建议

## 1. 结论

当前仓库结构已经能支撑开发，但存在三个明显问题：

1. **文档过度集中**：`docs/` 下同时放需求、设计、计划、日志、调研和临时产物，层次不清。
2. **运行时/调研数据混放**：`data/` 中既有种子输入，也有 SQLite 运行时文件和 12306 探测材料。
3. **后端职责边界开始膨胀**：`backend/app/services/` 同时承载搜索、缓存、遥测、静态库、时间扩展图和候选站生成，后续会继续变厚。

如果当前目标只是“先把结构理顺”，建议先做**文档层整理**，不动业务代码。

## 3. 已发现的结构问题

### 3.1 `docs/` 需要分层

现在所有文档都平铺在同一层，文件数量已经明显增多。继续堆下去会出现两个问题：

- 找文档困难，阅读顺序不明确。
- 新文档很容易命名混乱，最后变成“文档垃圾桶”。

### 3.2 `data/12306_query_train_info_probe/` 不适合放在运行数据目录

这个目录更像一次性调研/抓包/验证材料，不是稳定输入数据，也不是运行时产物。

建议至少把它从 `data/` 的“业务数据语义”里分离出去，避免后续误用。

### 3.3 `backend/app/services/` 过于宽泛

当前 `services/` 已经包含：

- 路线搜索
- 时间扩展图搜索
- 静态票价库
- 缓存
- 遥测
- 候选中转站生成

这不是立即错误，但会让模块边界越来越模糊。

### 3.4 前端主组件可能继续膨胀

`frontend/src/App.tsx` 已经同时承担页面状态、请求、流式消费和格式化逻辑。现在还能维护，后面会变重。

## 4. 整理优先级

### P0：先做安全整理，不动业务代码

建议先完成下面几件事：

1. 新增 `docs/README.md`，作为文档索引。
2. 新增 `.env.example`，集中列出 README 里散落的环境变量。
3. 补充 `.gitignore` 对抓包/探测类临时材料的约束，避免敏感或大体积文件混入。
4. 明确 `data/12306_query_train_info_probe/` 的归属：保留摘要还是迁移到 `docs/research/`。

### P1：重组文档目录

建议把 `docs/` 按用途分层：

```text
docs/
  README.md
  architecture/
  designs/
  plans/
  research/
  logs/
  writing/
```

对应建议：

- `architecture/`：项目整体架构、总体设计
- `designs/`：静态票价库、时间扩展图、票价获取方案等设计文档
- `plans/`：实施计划、修复计划、落地计划
- `research/`：数据源调研、探测材料摘要
- `logs/`：开发日志
- `writing/`：博客建议或对外输出草稿

### P2：重构代码结构

这一步会产生 import 和测试路径变更，建议最后做：

1. 后端把 `services/` 拆成更清晰的搜索域 / 持久化域。
2. 前端把 `App.tsx` 中的纯函数和子区域拆出去。
3. 每次只做一个切片，避免和业务功能混在同一个提交里。

---

## 6. 风险控制

1. **不要先做大规模移动**：当前工作区已有大量未提交修改，直接重命名/搬迁文件会扩大 diff。
2. **先补索引，再做迁移**：先让文档“可导航”，再考虑物理搬迁。
3. **抓包类材料默认脱敏**：如果保留 `probe` 内容，必须明确哪些可以提交，哪些不能提交。
4. **每次只整理一个层级**：先文档，再数据，再代码，不要一起动。

---

## 7. 最小可执行动作

如果只做一件事，优先级如下：

1. 写 `docs/README.md`。
2. 写 `.env.example`。
3. 评估 `data/12306_query_train_info_probe/` 是否迁移。
4. 再决定是否建立 `docs/architecture/`、`docs/designs/` 等子目录。
