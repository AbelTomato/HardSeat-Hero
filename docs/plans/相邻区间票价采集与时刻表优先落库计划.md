# 相邻区间票价采集与时刻表优先落库计划

## 1. 背景

当前全量快照采集目标是为时间扩展图搜索准备本地静态基础边，在线搜索阶段只读 SQLite，不访问 12306 远程接口。

前一版方案按每趟车全部 `i < j` 同车 OD 查询公布票价：

```text
平均每车 15 站
每车 OD = 15 * 14 / 2 = 105
约 10,000+ 车
票价请求量约 1,000,000+
```

实测 12306 接口吞吐后，该方案不适合作为第一阶段全量采集方案。

## 2. 已验证事实

### 2.1 接口频率测绘结果

使用脚本：

```text
backend/scripts/probe_12306_rate_limits.py
```

测试日期：`2026-07-12`。

稳定结论：

| 接口                   | 稳定 profile                  | 结果         | 备注                         |
| ---------------------- | ----------------------------- | ------------ | ---------------------------- |
| `queryTrainInfo/query` | `concurrency=4, delay=0~0.05` | 30/30 成功   | p50 约 29s，p95 约 56s       |
| `queryTrainInfo/query` | `concurrency=6/8`             | 开始失败     | 吞吐提升有限                 |
| `queryAllPublicPrice`  | `concurrency=4, delay=0~0.05` | 30/30 成功   | p50 约 25~33s，p95 约 49~58s |
| `queryAllPublicPrice`  | `concurrency=6/8`             | 失败明显增加 | p95 可超过 100s              |

推荐本地长跑参数：

```text
--train-info-concurrency 4
--train-info-delay 0
--price-concurrency 4
--price-delay 0
--timeout-seconds 60
--max-retries 1~2
```

### 2.2 只爬车次时刻表耗时估算

已有探测显示 `getTrainName(date)` 返回原始记录约 `16,000+`，按 `train_no` 去重约 `11,000~12,000`。

按实测稳定吞吐：

```text
train-info ≈ 0.46 ~ 0.54 requests/s
11,847 / 0.54 ≈ 6.1 小时
11,847 / 0.46 ≈ 7.2 小时
```

考虑失败重试和网络波动，实际估算：

```text
只爬时刻表：约 7~9 小时
```

### 2.3 旧离线数据不可作为当前日期数据源

已抽样检查：

```text
data/12306-code-database-2023-12-15-
```

该数据包含：

- `utf8/index/*.csv`
- `utf8/{C,D,G,K,...}/*.json`
- `db/train_basic.db`
- `db/station_based_code.db`

其中 JSON 结构与 `queryTrainInfo/query` 返回结构接近，可作为解析格式参考。但数据日期为 2023 年，不能作为 `2026-07-12` 的真实时刻表来源。

结论：

```text
旧离线库只作结构参考，不参与当前 2026 日期采集。
```

## 3. 核心策略调整

### 3.1 第一阶段：优先爬取并落库车次时刻表

新增脚本模式：

```text
--train-info-only
```

行为：

```text
1. 调用 getTrainName(date)
2. 按 train_no 去重
3. 并发调用 queryTrainInfo/query 获取经停表
4. 写入 train_service_snapshot / train_stop_snapshot 或等价快照表
5. 记录 train task 状态
6. 不执行 queryAllPublicPrice
```

目的：

- 先拿到指定日期的真实车次经停表。
- 后续票价任务全部由本地时刻表派生。
- 避免时刻表与票价采集耦合导致长任务不可恢复。

### 3.2 第二阶段：只生成相邻站票价任务

新增参数：

```text
--price-task-mode adjacent|all
```

模式说明：

| 模式       | 含义                     | 每车平均请求规模 |
| ---------- | ------------------------ | ---------------- |
| `all`      | 生成全部 `i < j` OD      | 15 站约 105 个   |
| `adjacent` | 只生成相邻 `i -> i+1` OD | 15 站约 14 个    |

第一阶段建议默认使用：

```text
--price-task-mode adjacent
```

估算：

```text
10,000+ 车 * 约 14 个相邻区间 ≈ 140,000 ~ 150,000 个 train adjacent OD
```

相比全部 `i < j`，请求规模降低约 7.5 倍。

### 3.3 第三阶段：对 station OD 去重

即使只采相邻区间，也应继续按 `(from_station, to_station)` 去重。

目标模型：

```text
Train adjacent PriceTask[]
  ↓
group by (from_station, to_station)
  ↓
每个 station OD 请求一次 queryAllPublicPrice
  ↓
返回 rows 后匹配该 OD 下多个 train task
  ↓
写入 train_od_fare_edge
```

原因：`queryAllPublicPrice(from, to)` 返回该 OD 下多趟车的公布票价，不是单车接口。

## 4. 票价语义与风险

### 4.1 相邻段累加不是严格单票公布票价

相邻区间票价可用于时间扩展图建边，但不能直接声明为同车长区间真实单票价格。

风险原因：

1. 铁路票价可能包含起步价、递远递减、取整规则。
2. 高铁、动车、普速、卧铺等席别计价规则不同。
3. `A -> D` 单票价格可能不等于 `A -> B + B -> C + C -> D`。
4. 相邻段累加更接近“分段购票成本”，不等同于“一张长区间票”。

因此，产品和数据语义应标记为：

```text
相邻段累加成本 / 图搜索近似边权
```

不得标记为：

```text
同车任意 OD 真实公布票价
```

### 4.2 必须抽样验证误差

新增验证脚本：

```text
backend/scripts/compare_adjacent_vs_direct_price.py
```

抽样逻辑：

```text
1. 从已落库时刻表中抽样 50~100 趟车
2. 每趟车选择若干 OD：
   - 相邻站 i -> i+1
   - 中距离 i -> i+3
   - 长距离 first -> last
3. 查询 direct OD 公布票价
4. 查询相邻段公布票价并求和
5. 按 train_no / train_code / seat_type 对齐
6. 输出误差报告
```

输出指标：

```text
sample_count
exact_match_count
diff_count
avg_abs_diff
max_abs_diff
by_seat_type_diff_stats
```

验证结论用于决定：

- 是否接受相邻段累加作为第一版成本。
- 是否对长距离 OD 增加直接票价补偿采集。

## 5. 云函数高并发方案

云函数可以作为后续扩展，但不应先于任务粒度优化。

### 5.1 不推荐直接 fan-out

不推荐：

```text
150,000 个请求一次性分发到云函数
```

风险：

- 12306 可能按 IP、TLS 指纹、请求模式限流。
- 云函数冷启动和超时会放大失败率。
- 失败重试不可控。
- 重复任务难以去重。

### 5.2 推荐队列式 worker

推荐结构：

```text
中心任务表 / 队列
  ↓
多个云函数 worker 分片拉取任务
  ↓
每个 worker 小并发查询
  ↓
结果写入中心数据库或对象存储
  ↓
失败任务进入 retry / dead-letter
```

任务粒度：

```text
unique station OD
```

而不是：

```text
train OD
```

### 5.3 吞吐估算

本地稳定吞吐：

```text
public-price ≈ 0.5 requests/s
150,000 / 0.5 ≈ 83 小时 ≈ 3.5 天
```

若云端稳定达到：

```text
10 requests/s -> 约 4.2 小时
50 requests/s -> 约 50 分钟
```

但该吞吐需要独立云端测绘确认，不能假设 12306 会接受。

## 8. 验收标准

### 8.1 时刻表阶段

- 能完整获取指定日期 train list。
- 能断点续跑经停任务。
- 单车失败不影响全局。
- 输出成功数、失败数、总停站数、耗时。
- 本地查询能读取落库的经停序列。

### 8.2 相邻票价阶段

- `adjacent` 模式任务量约等于 `sum(stop_count - 1)`。
- station OD 去重后请求数小于等于 train adjacent OD 数。
- 票价查询失败可恢复。
- 写入的边可被时间扩展图搜索读取。

### 8.3 误差验证阶段

- 输出 direct OD 与相邻累加的误差报告。
- 明确哪些席别误差可接受，哪些不可接受。
- 根据误差决定是否补采长距离 direct OD。

## 9. 当前决策

当前不采用 2023 离线库作为 2026 时刻表来源。

当前优先级：

```text
P0: train-info-only 真实时刻表落库
P1: adjacent 票价任务模式
P2: station OD 去重请求
P3: 相邻累加误差验证
P4: 云函数 worker 方案
```
