# 时间扩展图全量快照快速落地计划

## 1. 目标

本计划用于快速落地一个只依赖本地数据的时间扩展图搜索能力：

```text
一次性爬取或导入全量经停时刻 + 同车 OD 公布票价
  ↓
写入本地 SQLite 基础边图
  ↓
在线查询按用户出行日期动态展开时间扩展图
  ↓
本地 Dijkstra / label-setting 搜索返回方案
```

当前阶段优先满足“快速获取方案”，不追求完整铁路运营日历建模。

全量挂机爬取的数据源、任务状态、并发限速、重试和断点续跑设计见：[`docs/时间扩展图全量快照爬取设计.md`](./时间扩展图全量快照爬取设计.md)。

## 2. 明确范围

### 2.1 做什么

1. 新增一套不按 `service_date` 重复存储的基础边数据结构。
2. 支持从 CSV 导入一次性全量快照。
3. 时间扩展图搜索在线阶段只读本地 SQLite。
4. 查询时用 `query.date` 把基础边的相对时刻展开成实际 `datetime`。
5. 保留当前 `ROUTE_SEARCH_ENGINE=time-expanded` 切换方式。

### 2.2 不做什么

1. 不处理实时余票。
2. 不处理每日开行日历。
3. 不处理临时停运、加开、调图差异。
4. 本文不展开全国 OD 暴力采集实现细节；全量车次中心爬取方案另见全量快照爬取设计文档。
5. 不在在线搜索阶段调用远程接口。
6. 不触发 `static-with-remote-fallback`。
7. 不把同车长 OD 票价拆成相邻区间价格相加。

### 2.3 快速版本假设

为快速落地，做以下显式假设：

```text
导入到基础边表中的车次边，在用户查询日期默认可用。
```

这意味着搜索结果表达的是“基于一次全量快照的静态可行方案”，不是实时可购票方案，也不是严格运营日历方案。

## 3. 当前实现问题

现有最小版时间扩展图搜索读取：

```text
train_od_price_snapshot(provider, service_date, from_station, to_station, depart_at, arrive_at, seat_type, price, ...)
```

该表是“按出行日保存的 OD 边快照”。它适合快速验证算法，但不适合“一次全量爬取后长期查询”。

当前目标更适合基础图模型：

```text
车次 + 站序 + 相对时刻 + 同车 OD 票价
```

在线查询时再按用户日期展开为具体时间事件。

## 4. 目标数据模型

### 4.1 新增表：`train_od_fare_edge`

建议在 `backend/app/services/static_price_repository.py` 的 `_ensure_schema()` 中新增：

```sql
CREATE TABLE IF NOT EXISTS train_od_fare_edge (
    provider TEXT NOT NULL,
    train_no TEXT NOT NULL,
    train_code TEXT NOT NULL,
    from_station TEXT NOT NULL,
    to_station TEXT NOT NULL,
    from_station_no INTEGER NOT NULL,
    to_station_no INTEGER NOT NULL,
    depart_time TEXT NOT NULL,
    arrive_time TEXT NOT NULL,
    depart_day_offset INTEGER NOT NULL DEFAULT 0,
    arrive_day_offset INTEGER NOT NULL DEFAULT 0,
    duration_minutes INTEGER NOT NULL,
    seat_type TEXT NOT NULL,
    price TEXT NOT NULL,
    currency TEXT NOT NULL DEFAULT 'CNY',
    source TEXT NOT NULL,
    fetched_at TEXT NOT NULL,
    raw_hash TEXT,
    PRIMARY KEY(
        provider,
        train_code,
        from_station,
        to_station,
        seat_type
    )
);
```

索引：

```sql
CREATE INDEX IF NOT EXISTS idx_train_od_fare_edge_from
ON train_od_fare_edge(provider, from_station, price);

CREATE INDEX IF NOT EXISTS idx_train_od_fare_edge_train
ON train_od_fare_edge(provider, train_no, from_station_no, to_station_no);
```

### 4.2 字段语义

| 字段                | 语义                                              |
| ------------------- | ------------------------------------------------- |
| `provider`          | 快照来源，例如 `full-snapshot`                    |
| `train_no`          | 车次号                                            |
| `train_code`        | 车次代码；无差异时可等于 `train_no`               |
| `from_station`      | 乘车边起点站                                      |
| `to_station`        | 乘车边终点站                                      |
| `from_station_no`   | 起点站在该车次经停序列中的站序                    |
| `to_station_no`     | 终点站在该车次经停序列中的站序                    |
| `depart_time`       | 相对发车时刻，格式 `HH:MM` 或 `HH:MM:SS`          |
| `arrive_time`       | 相对到达时刻，格式 `HH:MM` 或 `HH:MM:SS`          |
| `depart_day_offset` | 相对运行基准日的出发日偏移，通常为 `0`            |
| `arrive_day_offset` | 相对运行基准日的到达日偏移，跨日到达为 `1` 或更大 |
| `duration_minutes`  | 乘车边耗时                                        |
| `seat_type`         | 席别                                              |
| `price`             | 公布票价                                          |
| `source`            | 数据来源标记                                      |
| `fetched_at`        | 本次快照获取时间                                  |

## 6. 搜索展开逻辑

### 6.1 读取基础边

`TimeExpandedRouteSearchEngine` 不再只读：

```python
query_train_od_prices(provider, service_dates)
```

而应优先读取：

```python
query_train_od_fare_edges(provider)
```

### 6.2 日期展开

基础边只保存相对时刻。查询时动态展开：

```python
base_date = query.date + timedelta(days=base_day_offset)
depart_at = combine(base_date + timedelta(days=edge.depart_day_offset), edge.depart_time)
arrive_at = combine(base_date + timedelta(days=edge.arrive_day_offset), edge.arrive_time)
```

为了支持跨日换乘，第一版可对基础边展开 3 天窗口：

```python
for base_day_offset in range(3):
    materialize_edge(edge, query.date + timedelta(days=base_day_offset))
```

含义：假设基础边中的车次每天都可用，并为查询日、次日、第三日各生成一组时间化边。

### 6.3 过滤规则

沿用当前时间扩展图搜索规则：

1. 初始状态在 `query.from_station`。
2. 第一段允许从查询日期窗口内出发。
3. 后续段必须满足：

```text
edge.depart_at >= state.arrive_at + query.min_transfer_minutes
```

4. 换乘次数满足：

```text
len(segments) - 1 <= query.max_transfers
```

5. 到达 `query.to_station` 后构造 `TransferPlan`。
6. 按最低总价排序，截断前 10 个方案。

## 9. 风险和后续演进

### 9.1 风险

1. 默认所有导入车次每天可用，会产生运营日历误差。
2. 一次性快照可能过期。
3. 如果全量数据源本身缺边，搜索仍会返回空方案。
4. 数据量很大时，直接全表读入内存需要优化为按站点索引或懒加载。

### 9.2 后续演进

当前快速版本跑通后，再考虑：

1. 增加开行日历表。
2. 增加快照版本号和有效期。
3. 增加按 `from_station` 懒加载边。
4. 增加图索引缓存。
5. 支持增量更新。
